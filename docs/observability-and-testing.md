# Observability and testing

## Observability

A console — a web client rendering the agent's event log — answers the question that is otherwise pure guesswork: "what was the agent thinking." It is not a bespoke query UI over the server: it is a **materialising read replica**. It obtains the event log from a source, folds it through the agent's own [materialiser](events-and-storage.md#materialised-graph) compiled to WebAssembly (real Rust, real SQLite bundled into the module), and renders every view as a query against that local graph. Because the reconstruction layer *is* the agent's projection code rather than a parallel reimplementation, no view can drift from what the agent actually computed.

### Three audiences shape the design

You during development (everything, fast); future-you investigating an incident (reconstruct past state); and the agent itself (introspection over the same surface, a standing design pull rather than a built feature). The first two are what the console serves. The third pull follows from the read views being a pure fold of the log: the machinery that answers "what did the agent know at seq N" is the same machinery an agent could turn on itself.

### Access model

The console is a separate client, not a second connection into the store. It reads one thing — a stream of events — and reconstructs everything else locally, so a package on disk and a running agent are the same client over two sources (see [One client over several sources](#one-client-over-several-sources)). A live agent is reached over its `/control` surface, authenticated by the per-surface trust model (a loopback peer is trusted and needs no key; a remote peer presents the API key — see [Clients and the server boundary](trust-and-authority.md#clients-and-the-server-boundary)). The on-box operator uses the same event-stream path as the remote one, so there is no separate read-only SQLite path to special-case.

The replica never mutates state locally, so the audit trail is unbroken. Every operator write — deleting a test memory, resolving a cross-platform identity merge, conversing as a participant — is a `POST` to `/control` that lands as a normal event tagged `source: Operator`, exactly as an agent write does; the console then sees the resulting event arrive on its tail and re-renders. Operator writes are therefore available only against a live agent with the cursor at the head; a scrubbed-back graph and a loaded eval package are read-only history.

Cross-platform identity merges live in the **Relations** view: the folded log surfaces every merge proposal with its status (pending or merged), derived by running a union-find over the identity edges, and — against a live agent at the head — the operator confirms each one they are persuaded by; an unconfirmed proposal simply stays pending. Asserting a fresh `same_as` is the same kind of operator event. The read-only eval viewer shows the proposals and their standing but offers no resolve control, since it cannot author an operator event.

### The views

The console is organised into **frames** (application shells) that share a common set of timeline-scoped **views** over one event stream. The eval frame drives the views off a run's embedded log; the agent frame drives the identical views off a live `/control` tail whose head grows as events arrive. The active view and a timeline cursor both live in the URL, so browser back and forward move between views and timeline positions the same way in either frame, and a global scrubber at the foot folds the graph to any `seq` before the views query it.

Seven views hang off that shared timeline:

1. **Conversation** — "what was the agent thinking," made literal. For every conversation in the stream (reachable from a sidebar): the participants, the assembled brief and system prompt at the start, and every turn. Per agent turn it shows the deliberation — each step's reasoning, the Lua executed, and what came back — with aborts and errors rendered distinctly from successful blocks. Each model call within a turn carries a **per-call context debugger** (see below). Against a live agent the composer at the foot lets the operator speak into the conversation as a participant.
2. **Background** — the background passes' log-only audit events (description regenerations, belief arbitrations, and inferred links), grouped by pass type. These events carry no turn attribution — they run asynchronously, often long after the turn that inspired them — so they surface as a top-level timeline rather than being mis-attributed to a turn, each row annotated with the conversation turn that last touched the memory before the pass ran.
3. **State** — the materialised graph as it stands at the cursor. Browse by namespace, tag, or recency; open a memory to see its contents (with `told_by` and visibility), tags, in and out links, description, per-memory history, and its `same_as` class. The console deliberately bypasses the visibility predicate and the live filters — superseded entries and soft-deleted memories render, marked as such — because history surfaces are the point of an operator tool.
4. **Relations** — the graph read by edge rather than by node: the relation registry as a filterable table, a force-directed graph below it, and a linked-pairs list beneath that. Each relation has a stable colour hashed from its name. The `same_as` collapse (on by default) runs a union-find over the identity edges *before* relation filtering, so a merged class renders as one node with a member-count badge that expands to its members. The merge-proposal confirmation surface, where the operator confirms the agent's pending proposals, sits at the top.
5. **Agenda** — the agent's horizon: its upcoming dated and recurring events, soonest first, projected forward from "now" at the cursor (recurring rules to a 60-day horizon, one-offs unbounded). It is the calendar face of the recurrence the State view marks per memory, and it time-travels with the scrubber.
6. **Events** — the log itself, filtered by time (the cursor), by type (category groups plus click-to-filter on an exact type), and by target or participant (a memory's page jumps to "all events touching this"). An eval run's step journal renders as markers over the stream. Filtering by `source` is not offered (see [Known limitations](limitations.md)).
7. **Time-travel** — diff the materialised graph between two points in the log. The global scrubber sets one end; a baseline slider sets the other; the two are compared forward regardless of order, off a private replica so the comparison never disturbs the shared timeline. What changed is read off the graph — memories added or removed, entries or descriptions or tags or renames gained — with the bridging events listed beneath.

The agent frame appends three live-only tabs that are not timeline-scoped (so the scrubber steps aside while they are open):

- **Console** — an operator Lua console: an editor that runs ad-hoc Lua against the agent's live graph in a no-commit sandbox. This is the agent's *real* async Lua, run server-side (`POST /control/lua`, a read-only dry-run against live head), with the agent's own `memory.*` / `tags.*` / `links.*` / `calendar.*` surface standing alongside as a filterable reference. Reads see real memory; nothing the block writes persists. MCP is off unless the operator opts in, since a tool call is a real external effect even in the sandbox.
- **Prompts** — the prompt-template registry as recorded in the log.
- **Settings** — the instance's settings, editable against a live agent.

Two more surfaces sit outside the shared timeline, in their own frames:

- **The eval frame's chrome** — a scenario overview (per-scenario verdicts, judge rationales, pass rates, and latency and token metrics) and a per-run frame that wraps the shared views in a scenario/run header with the run's oracle verdicts.
- **Trends** — a page over the tracked `eval/history.jsonl` trend record, plotting metrics across runs over time.

One built bundle serves every host; the serving binary announces its mode at runtime (`window.__APP_MODE__`, injected into the HTML shell at serve time). The agent serves the embedded live view at its root; the eval binary serves the console pointed at its own same-origin live eval stream; a standalone build is the full console with a landing screen for opening a package file, a history file, or a live connection.

**Brief trace.** The brief composer computes negative space — which memories it considered, which it filtered and why — and the console reconstructs it *without a logged trace event*. Because composition is a pure function of the graph, the present set, and the clock, and the replica holds all three (fold to the session's seq; the present set and working set from the `SessionStarted` payload; the time from the session start), the wasm wrapper re-runs a trace-returning `compose` variant and reconstructs why each memory did or did not reach the brief: surfaced, passed the predicate but trimmed by recency, or filtered by a visibility verdict (with the reason). The brief settings are folded from the log at the same horizon, so the re-derivation reproduces what the server composed. This is the record-facts-derive-judgements discipline: the fact (the session's inputs) is logged, the judgement (the trace) is recomputed on demand, so no `BriefTrace` event bloats every session. The one caveat is that re-derivation runs *current* composer logic — exact for the eval viewer and a live-recent agent, and carrying a "logic may have changed" asterisk for cross-version time-travel.

**Per-call context debugger.** Each model call the loop makes carries a context breakdown, reconstructed from the recorded `ModelCalled` event. A one-line heading reports the measured prefix-cache warmth, the tokens in and out, the re-prefill cost (prompt tokens minus cache-read tokens), the compaction fill against the budget, and the digest verification. Expanding it shows a stacked bar whose segments are the prompt's parts in send order. Below the bar sits one row per part — the system prompt's sections, the tool set, and the message history — each carrying a token count, a provenance badge (recorded, or re-derived and marked inferred), and its percent of the context window, and each expandable to the exact content it measures. When the prefix cache broke, the debugger names the culprit row (the brief that re-froze, the tool set that changed, or the rebuilt history). The denominators (budget and context length) are the ones in effect *when the call was made*, so a later settings change never repaints an earlier call's numbers. The turn's footer reuses the same display for its final call, aggregating the whole turn's generation across its steps. The same measured warmth rolls up across a set of calls — the median warmth and the total tokens the provider re-encoded fresh (prompt minus cache-read) — into the conversation header, the eval run frame, and the eval scoreboard, so a conversation or scenario that thrashes the prefix cache is visible at a glance; a median below 90% is marked in clay, the warning accent. The aggregate reports `unknown` rather than a fabricated number when no call in the set carried both counts.

**Model-interaction record.** The context debugger and the deliberation view read a log-only `ModelCalled` event emitted for every model call — each step of a turn and the post-turn describe synthesis. It carries the request, the model's `reasoning` (the serving layer's `reasoning_content`, when present), the parsed completion, the `finish_reason`, the full token usage, and the call's wall-clock latency (`duration_ms`). Block execution time rides on the existing `LuaExecuted` (`duration_ms`). Together with the per-call latencies and the events' `seq` ordering, the console reconstructs the whole turn timeline and reads off where the time went — distinguishing a slow inference step from a slow tool call, or a model cold-start from genuine think time. The request is stored as a delta to keep the log from ballooning: the first call of a `(turn_id, phase)` group records a `RequestRecord::Base` (the frozen system prompt, tools, tool choice, thinking flag, and the initial buffer) and each later call a `Continuation` carrying only the messages appended since. A full prompt is reconstructed by walking the group and concatenating the deltas, then re-hashed and checked against the `Sha256` `request_digest` every record carries. The wasm wrapper does this walk itself (`requestDigests`), classifying each call `verified`, `mismatch` (the reconstruction diverged and must not be trusted silently), `unverifiable` (a structured `Synthesis` call carries a `response_format` the record does not store, so its digest cannot be reproduced), or `unrecorded` (its base fell outside the loaded window). Capture is operator-tunable via the `CaptureLevel` setting — `Full` stores the delta and digest, `Digest` keeps only the digest and the full response, `Off` records nothing — defaulting to `Full`; the non-deterministic deliberation (reasoning, finish reason, usage, latency) is captured at every level above `Off`. The record is **log-only telemetry**: the materialiser ignores it, so it never enters the graph projection — under [faithful replay](events-and-storage.md#faithful-replay) the reconstructed state is identical with or without it, and the recorded reasoning, usage, and latency are reproduced verbatim because replay reads them rather than recomputing.

**Agent creation lives here too** (see [Initialisation and lifecycle](lifecycle.md)): you can watch the genesis events stream into the event view as the agent is born, and an agentless instance is gated behind a genesis screen in the agent frame until it is created.

**Runtime metrics.** A Prometheus-scrapable `GET /control/metrics` endpoint gives an operator — or a Grafana dashboard pulling on an interval — the four golden signals for the turn path (throughput, latency, errors, saturation) plus agent-state gauges and the agent's outward I/O, so "is the server up" is separable from "is the agent healthy" from "where did the time go." Throughput counters cover turns, model calls, MCP tool calls, Lua blocks, memory searches, wake-ups fired and surfaced, compactions, flush turns, sessions opened and closed, and deferred turns. Latency histograms — turn, model call, MCP call, and memory search — share one bucket mesh so they stay comparable. Errors are labelled by category (turn, describe, checkpoint, indexer, scheduler, sweep) and cause (model, lua, store, graph; `none` for non-turn errors), alongside model-transport health counters (retries, circuit-breaker fast-fails, and a circuit-state gauge). Saturation covers model prompt and completion tokens. Gauges, refreshed at scrape time from the instance's current state: process uptime, the event-log head seq and store size in bytes, live sessions, graph counts (memories, entries, links, tags, relations), worker lag (how far the indexer and describer trail the log head), and MCP health (servers up, projected tools). Built on the `metrics` crate with a process-global recorder installed at boot. A recorder-install failure is non-fatal — the server serves on, and the endpoint answers `503`. The endpoint lives under `/control/*` (operator authority), and the metrics carry only structural counters and gauges — never conversational content.

**Per-turn span.** Each conversational turn runs under a `tracing` info span that records structural identifiers and informational metrics — the conversation, the agent's turn id (the durable key for finding the turn's events in the log), the outcome as a label (`reply`, `silent`, `max_steps`, or `error`), the wall-clock duration in milliseconds, and the step and block counts — but never conversational content. The outcome is a label, not the reply text, so the trace stays free of what was said. `init_tracing` enables `FmtSpan::CLOSE` so the span's close carries the turn's total duration.

**Boot log.** At startup a single `tracing::info!` records the resolved storage directory, the bind address, the model and embedding endpoints (host and model id — keys are never logged), the genesis status, and each MCP server's projected tool count — the one-line answer to "is the server up and talking to what."

### One client over several sources

The eval viewer and the live console are *one client over several sources* — a static package file, a live SSE stream, or the agent's own log — so rendering an event log into the views above is the console's whole job. One such source is an **eval package** (see [Validation and the eval harness](#validation-and-the-eval-harness)), which embeds each run's event log. A source is anything that delivers the log: a package's embedded per-run `Event[]` loaded from a file, a live agent's `/control` event tail, or the eval harness's own live SSE stream (`GET /eval/stream`, a snapshot of the current package followed by live deltas folded into a growing package while a run proceeds). All feed the same wasm materialiser; the views query the resulting local graph and never know which source fed it. Time-travel therefore works against a live agent exactly as it works against a package — fold to seq N locally, diff two folds locally — with zero server involvement.

The materialiser is compiled to `wasm32-unknown-unknown` with SQLite bundled in, wrapped by the `console-wasm` crate (`crates/console-wasm`) into a `wasm-bindgen` surface: raw bytes in (the package is parsed with serde inside the module, one copy across the boundary), `apply`/`fold_to` for incremental feeding and time-travel, and purpose-built query methods per view returning values typed by the ts-rs bindings. Compiling the real Rust in buys two things a TypeScript reimplementation could not: the console renders the same fold the agent lives in (every new event type or payload version is handled once, in Rust), and any deterministic derived computation can be *re-run* rather than logged (the regenerative side of [faithful replay](events-and-storage.md#faithful-replay)) — the load-bearing cases being the brief trace and the system-prompt section provenance above.

The contract between the agent and the client is the **event log's types**, generated two ways from the Rust definitions: the ts-rs TypeScript bindings (`console/packages/wire/types/`) and the wasm materialiser bundle (`console/packages/wire/wasm/`). Both are generated artefacts, regenerated automatically by `cargo build -p zuihitsu` via the `console` Cargo feature whenever a wire type or the materialiser changes, so the Rust side stays the single source of truth. The types live in `zuihitsu-frontend-types`, a crate that depends only on `zuihitsu-core` to avoid a build cycle with the main crate's `build.rs`.

### Aesthetic

The look is **Japandi** — the quiet meeting of Japanese and Scandinavian design: a warm neutral ground (off-white, oat, undyed paper), sumi-ink text, and one or two muted, earthen accents (clay, sage) used sparingly; a real typographic scale doing the structural work; hairline rules and generous negative space in place of boxes and shadows; restraint as the organising principle. A console is read for hours, so it should be calm and legible — craft over chrome. It is expressed as design tokens (CSS variables wired into the Tailwind theme), not ad-hoc utility classes, so the system stays coherent, and the same tokens carry forward to every operator-facing surface.

## Testability and abstraction boundaries

Every external dependency and every stateful surface sits behind a trait, so a complete agent can be constructed in-memory for tests without standing up the model servers, a real database, the network, or a wall clock. This is a hard design requirement, not an aspiration: the model-free validation tiers below are only runnable cheaply if the substitution points exist, and the reply-lane eval harness reuses the same seams to boot a fresh agent per run.

The seams that are abstracted, each with a production implementation and a test fake:

- **Model client.** The inference interface is the `ModelClient` trait (`generate`), with the model's identity behind `model_id()`. Tests supply `ScriptedModel`, a deterministic fake that pops predetermined completions in order and records the `messages` and `tool_choice` each `generate` saw (so a test can assert what the model was shown — e.g. that a later turn replayed the live buffer, or that the loop withdrew the tools on its final step). `FlakyModel` wraps it to fail a programmed number of leading calls with a backend error before serving scripted completions, driving the transport-resilience paths (retries, the circuit breaker, deferred turns) without a real endpoint; it is distinguishable from `ScriptedModel`, whose only failure is the test-logic `Exhausted`. An agent-level scenario is thus deterministic and needs no GPU.
- **Embedder.** The `Embedder` trait (`embed`) is its own seam, separate from the model client, since embeddings run against a different endpoint. `FakeEmbedder` returns fixed-dimensional deterministic vectors, so a retrieval test exercises the index without a real embedding server.
- **Clock.** `now()` reads an injected `Clock` trait. `SystemClock` is the production wall clock; `ManualClock` is advanced explicitly (`set`, `advance_millis`) and, because a clone shares the same underlying atomic time, a clone handed to the system under test and a clone retained by the test observe the same advances. Tests drive temporal logic, calendar windows, recency decay, and scheduled wake-ups without real time passing.
- **Storage.** The event log is the `Store` trait: a single-writer, append-only log preserving one total order over `Seq`. `MemoryStore` serves tests, the no-I/O path, and the wasm replica; the durable file-backed SQLite backend lives in the main crate, since it needs the host filesystem. The materialised graph is `Graph`, opened with `Graph::open_in_memory()` for tests; the vector index is the `VectorIndex` trait, with `SqliteVectorIndex::open_in_memory(dimensions)` for the in-memory `sqlite-vec` case. A test builds a log, materialises it, and inspects state with no files. Faithful replay falls out of the store seam: read from `Seq::ZERO` and events come back in commit order, so the system's behaviour is a pure function of the event log modulo declared nondeterminism (ULID minting, wall-clock stamps).
- **MCP client.** The outward I/O boundary is a pair of traits: `McpHost::spawn(name, config) -> Box<dyn McpInstance>` brings a server up, and `McpInstance` exposes it — `tools()` (the catalogue snapshotted at spawn), `call(tool, arguments) -> Result<McpOutput, McpError>`, and `shutdown()`. Abstracting at the instance level makes lifecycle testable, not just calls. The fake, `FakeMcpHost` (scripted via `FakeServer`), is scriptable: a canned tool catalogue, canned per-tool results (text, non-text blocks, `isError`), injected latency, a server that spawns dead, and a spawn failure — so a scenario exercising `mcp.<server>.*` needs no real subprocess or network. The env values a real server reads are redacted intrinsically at the config's serialisation boundary, so the config view cannot leak a secret.

With these in place, a test seeds an event log (or drives the imprint or conversation loop through a scripted model), materialises, and asserts — on the resulting state, on what entered a brief, or on what the agent said. Two validation lanes fall out of this and are described next: a **model-free** lane of ordinary `cargo test` unit tests that assert directly on the visibility predicate (`visible(...)`) and on the composed brief, and a **reply** lane — the standalone eval harness — that drives the real loop against a live model with a `ManualClock` and inspects what the agent actually said. Both boot the agent through the same seams; only the model and the assertion surface differ.

## Validation and the eval harness

The abstraction boundaries (see [Testability](#testability-and-abstraction-boundaries)) make validation runnable tests rather than a hand-checklist. Validation splits into two lanes by what a check needs — a **model-free** lane of ordinary unit tests and a **reply** lane run by a standalone eval harness — because the questions they answer are different in kind: mechanism is exact and cheap, judgement is stochastic and expensive.

### The model-free lane: predicate and brief

Two assertion surfaces are pure, deterministic, and model-free, so they run under `cargo test` on every change:

- **Predicate** — assert directly on `visible(entry, present_set)`. Deterministic, model-free, microseconds. This is where mechanism lives: the subject-guard, `Exclude` resolution, class-aware presence, and the write-time defaults. Most mechanism scenarios are here.
- **Brief** — build the contextual brief for a present set and assert that a fact is present or absent, and that a teller-private fact's inline marker carries its `told_in` room and the `#confidential` flag. Also deterministic, because brief composition is deterministic (the [determinism principle](overview.md#architectural-principles)). A leak into the brief is a mechanism bug catchable here without spending a single inference.

Both run against materialised state built from a seeded event log through the in-memory backends, so they exercise exactly the pure function the store seam defines. Pushing these through a real model would make exact checks flaky for no benefit and burn inference on questions a predicate answers precisely. The model-free lane catches everything it can, and the reply lane spends the model only where judgement is genuinely under test.

### The reply lane is a standalone harness that emits an eval package

The reply lane is run by a standalone **eval harness** (`crates/eval`) — a separate binary over the library, kept out of `cargo test` because it drives a live local model and so needs a configured endpoint and a GPU. It boots a fresh agent per run through the same seams the unit tests use — an in-memory store, an in-memory graph, an in-memory `sqlite-vec` index, real embeddings, the real model, and a `ManualClock` anchored at a fixed start (`run_start()`, 2026-06-08T00:00:00Z) so scenario timing is reproducible. Time is controlled; memory and inference are exercised for real. A scenario seeds deterministic setup state by emitting events straight into the in-memory log (fast, no model), then drives turns against the live model.

**A scenario is a declarative script, not an imperative body.** `fn steps(&self) -> Vec<EvalStep>` returns pure data the harness's executor interprets against the booted agent; `async fn assess(&self, events, judge) -> Vec<Verdict>` judges the resulting log. Keeping the two separate is what lets a stored package be re-assessed without re-running the model. The executor journals each step's event-log coverage — the seq span it appended and the log head after it — so a recorded run carries its own scenario↔log correspondence, which is what lets a run be replayed or resumed against the current script later. A scenario also declares the `InstanceFeatures` its agent is born with, narrowing the Lua surface so a behaviour can be tested in isolation (e.g. disabling `linking` so the link-inference pass is the sole path to a link), and whether it `needs_retrieval` or `needs_mcp` so the harness skips it with a warning rather than running it blind when no embedding endpoint or test MCP host is configured.

**The package's payload, per run, is the run's actual event log** (`Vec<Event>`). That makes an eval package a special case of the console's input — a live agent's log is the other — so the same event-to-view reconstruction renders an eval run and a live conversation alike (see [Observability](#observability)). The harness adds only the per-run **verdicts** and computed **metrics** (model calls, step count, model-call latency, token usage, all summed from the run's own `ModelCalled` events), plus one truthful cost the events cannot report: `wall_clock_ms`, the measured time to drive the run including the off-hot-path synthesis (the describer's work, the indexer) that records no `ModelCalled` and so is invisible to `total_latency_ms`. Runs are independent (a fresh agent each), so they parallelise across a bounded pool sized to the serving endpoint's real concurrency; the default is one, because a local endpoint serialises inference and a second in-flight run only contends. As each run completes it is folded into the growing package and streamed live.

### Stochastic assertions are asymmetric and N-run

A scenario runs N times (default 8, tunable per run). Sampling is at the model's normal settings rather than greedy decoding: the reply surface is a rate over N by design, so suppressing variation buys nothing — and the inference backend is not bit-deterministic even at temperature 0 regardless. The bar is the rate, not any single transcript, and the two bar kinds are asymmetric:

- A **gating** bar is for a must-not-surface property (a leaked confidence, catastrophic and rare). At its default `min_rate` of 1.0 it is the one-slip discipline — a single regression across N fails the harness with a non-zero exit, so the lane can stop a release. It also carries a tolerance: a model-judgement gating behaviour with a known error band can be gated below 1.0 (link inference, say, at 0.85), where an occasional miss is expected yet a systematic slide must still fail the run. At the default tolerance the boolean gating signal decides (so packages predating the rate field judge correctly); below it, the held rate of gating verdicts is compared against `min_rate`.
- A **metric** bar is for a should-mark or should-surface property (chronic, judgement-quality). It is reported as a rate against a threshold but never fails the run — a low rate is load-bearing news for tuning, not a stop.

An infrastructure failure (the model dropped, say) is recorded as a failed *metric*, not a gating leak: a flake must lower the rate without being reported as a safety regression.

### Assessment is a pure, re-runnable function of the log

A scenario's oracle reads the run's event log plus the scenario's own expectations. Deterministic properties are checked structurally, directly against the events (did a memory land at what visibility, was a tag applied, did the confidence stay attributed to a single teller); paraphrase-sensitive properties go through a judge. Because assessment takes the *stored* log rather than a live run, a package can be re-assessed without re-running the model, so an oracle or a judge prompt can be iterated cheaply against an existing corpus of runs (the `replay --mode rejudge` path). A judge error is never a harness crash — it becomes a failed verdict carrying the error, so a flaky judge call lowers the rate rather than aborting the run.

**The judge is the model, run clean-room.** Paraphrase-aware matching — a real model says "on his way out" for "being managed out," so a substring check silently passes a real leak — is delegated to the same model acting as judge, but in a fresh request sharing no context with the agent turn, fed a reprojection of only the slices relevant to the criterion (the confidence that was told, the reply under judgement), so it cannot rationalise from the agent's own reasoning trace. The whole reply is constrained to a structured `verdict` schema via `response_format`, and the verbatim completion is recorded in the package alongside the parsed verdict, because the matcher is a thing to review, not trust. A must-not-surface leak probe is run conservatively: it re-runs the "does this convey the fact?" question three times and treats any single "yes" as a leak, one-sided by design: it drives false negatives — the dangerous error for a safety gate — toward zero, while the judge's continuous-batching flicker lands only on safe false positives. A batched classifier keeps judge cost bounded where several turns need the same language judgement (the write-confirmation honesty check classes every non-committing turn in one call).

### Storage: a large gitignored package and a compact tracked history

A full package is large (every run's whole event log), so it is a local, gitignored artefact (`eval/<name>.json`) opened in the viewer for deep inspection, with a resumable `.jsonl` sidecar beside it so an interrupted run can be continued driving only the runs it does not already hold. A compact, append-only history file (`eval/history.jsonl`) is tracked instead: one deterministically-serialised line per run, keyed by the run's name so it correlates back to its package, carrying the git state it ran at, the model, and, per scenario, the bar it was judged against, the per-scenario rate, gating outcome, latency, token means, step median, and the per-criterion pass tallies — so trends over time stay visible, and a regression is traceable to the criterion that moved, without bloating the repository. Rates are rounded so an unchanged result produces an identical line.

### Live serving and replay

The run serves itself live over SSE by default (at `127.0.0.1:7878`) so the console watches the deliberation unfold — each run's log is polled between turns and new events streamed as they land, and the scoreboard fills in as runs complete — rather than seeing only the pass/fail at the end. Beyond `run`, the harness reads and re-drives a written package: `analyze` prints a per-scenario summary (with deltas against a baseline, or the failed runs' full deliberation traces), `events` lists a run's log grouped under the journal steps that produced it, and `replay` either re-judges a recorded package against the current criteria without re-running the model (`rejudge`) or rewinds one run to a chosen journal step and redoes the rest of the scenario live from that point (`resume`). Replay never writes trend history.

### The harness is also the backstop for materialiser logic bugs

This is not just for predicate bugs. Replay cures a corrupt graph but reproduces a buggy handler faithfully (see [Storage → Materialised graph](events-and-storage.md#materialised-graph)); since the model-free predicate and brief tests run against materialised state, a visibility-handling regression in a `(type, version)` handler fails them. That is the second failure class the model-free tier defends against, and the reason validation asserts on materialised output rather than on the predicate in isolation.
