# Visibility

Visibility is what reaches the model — the deterministic `visible(...)` predicate, the property the privacy guarantees rest on. This is the hardest correctness concern in a multi-participant memory, and the place to spend the most care.

The framing that organises the rest of this document: the agent is not a store with an access-control list; it is a node in successive information flows, and each surfacing is a new flow whose appropriateness is judged against the flow the information came in on. (This is an operationalisation of *contextual integrity* — privacy as appropriate flow over sender, recipient, subject, type, and transmission principle — though the spec needs none of that vocabulary to be implemented.) An ACL framing would get this wrong, because the subject-guard is the case access control structurally can't express: a fact that flows to everyone except its own subject, where in any ACL the subject would have read access to their own record. The suppression is a fact about the relationship among [teller](data-model.md#contententry), subject, and recipient, not about S's authorisation: the confidence was shared under a norm that the flow "aside-about-S → S" would break. The variants below are refinements of the same idea: `Exclude` narrows the permitted recipients; `told_in` plus `#confidential` carry the originating context so a cross-context surfacing can be recognised as one; and the teller-private marker is the honest admission that for the unnamed-third-party case the norm is under-determined, so no mechanism encodes it.

Every `ContentEntry` carries `told_by` and `visibility`; every `Link` carries `told_by`, `told_in`, and `visibility`. The filter is applied during [brief](conversations-and-briefs.md#contextual-briefs) composition, during search, on the agent's direct reads of a memory by handle, and at every link-reading surface (the brief's `<relationships>` block, the link readers, `mem:details`, and search salient relations); the agent never relays an entry or a relationship it shouldn't, through any channel. The predicate lives in one module (`crates/core/src/visibility/`) that every surface calls, so a single implementation governs all of them — it is deliberately I/O-free, taking an injected `class_of` resolver (a memory's [`same_as`-class](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated) id) so it stays a pure function the graph backs at the call site and the tests exercise directly.

## Visibility versus disclosure, and three postures

Two layers are easy to conflate, and keeping them apart is what makes the rest tractable. *Visibility* is the deterministic `visible(...)` predicate — what reaches the model, the property the privacy guarantees rest on. *Disclosure* is what the agent then chooses to say in its reply — a model judgement, as it is for every fact. A fact reaching the model is not the same as the agent volunteering it; the predicate governs the first, the agent's judgement the second. Spend the determinism budget on the first, where a guarantee is possible and matters, and let the second be judgement, where it is irreducible.

That split is why a fact about a third party comes in three postures, not two — because "an absent colleague mentioned Dave is a product designer" and "an absent colleague confided Dave is interviewing elsewhere" are not the same kind of thing, and collapsing both to one private default makes the agent useless at the first to stay safe on the second:

- **Confidence** (`PrivateToTeller`, or `Exclude` for a named carve-out) — a genuine secret. Hard-gated at the visibility layer: it reaches the model only when the teller is present and never in front of its subject (the subject-guard). Disclosure to any other co-present third party is a flagged judgement call. This is the tier the deterministic guarantee protects, and the only tier where "the audience isn't cleared" hides the content outright.
- **Attributed** — an ordinary fact learned secondhand: Dave's role, where he works, a preference a colleague relayed. Visible to anyone (it survives the teller's absence, so the agent can answer "what's Dave's role?" months later in a different room), but it reaches the model carrying a *provenance marker* — `[via Marcus]` — and is never distilled into a description, so it always surfaces as "Marcus told you this," never laundered into markerless prose. The agent applies ordinary social judgement before volunteering it; nothing about it is hidden, because nothing about it needs to be. `Attributed` is a nullary variant, not `Attributed(teller)`: the provenance it marks is the entry's own `told_by` (and `told_in`), carried by every entry regardless of visibility, exactly as `PrivateToTeller` reads its teller from `told_by` rather than duplicating it. `Exclude` carries a set inline because *that* data — the named third parties — exists nowhere else on the entry; the teller does, so the posture references it rather than copying it.
- **Public** — self-disclosed ("I'm Dave, a product designer"), announced, or about a thing rather than a person. Freely shareable, distilled into the description, no marker.

The agent classifies a write into one of exactly three choices — `public`, `attributed`, or `private` (`PrivateToTeller`) — through the `visibility` option on `mem:append` and `links.create`, and the [scaffold](conversations-and-briefs.md#system-prompt) teaches these three. The line between **confidence** and **attributed** is the one the agent draws at write time, from the framing — a hushed register, "between us," an asymmetric aside about someone absent, or a `#confidential` room mark all push toward confidence; a neutral "FYI" about a colleague's role is attributed.

The **conservative default is the floor**: a `person/*` fact a participant relays about someone else, left unclassified, stays `PrivateToTeller`. A missed call fails *closed*, toward privacy, never open — we never flip the default to public, because a confidence silently defaulting shareable is precisely the leak this document exists to prevent. The floor is not a verdict, though: the agent upgrades an ordinary fact to an explicit posture, so safety does not cost the agent its memory of everyday things.

**`Exclude` is read-only.** The fourth `Visibility` variant, `Exclude(set)`, is fully honoured at *read* time — the predicate, the markers, and the console displays all handle it, and the visibility tests exercise it — but no write surface constructs one: the `visibility` option and the scaffold offer only the three postures, and no operator or adjudication path emits an `Exclude`. The named-carve-out posture is thus complete read-time machinery; the passages below that describe `Exclude` as "the lever the agent uses" describe how it reads when one is present (see [Known limitations](limitations.md)).

## Superseded entries are not live

Alongside the visibility filter, all live surfaces — `visible(...)`, the brief's recent facts, and search — exclude any entry with `superseded_by` set, exactly as they exclude soft-deleted memories. The live entry reads exclude these in SQL (`class_entries`/`entries_local` carry `superseded_by IS NULL`), and the predicate's own first branch returns a `Superseded` verdict, which covers the search path — that path resolves a vector hit through `entry_by_id`, which does *not* filter, so the predicate is the guard there. This matters specifically because entries are embedded once on append and never re-touched (see [Storage → Vector store](events-and-storage.md#vector-store)): a corrected or retracted confidence stays semantically retrievable forever, so without this filter a superseded private aside could resurface through search even though a newer entry replaced it. Superseded entries remain visible only where history is the point — `mem:history()` and the console, which read the unfiltered `entries_local_history` primitive — deliberately bypassing the *supersession* exclusion. (They do not bypass the audience predicate below: `mem:history()` still withholds a superseded confidence from a present outsider, so "show the old value" never becomes a way around visibility.)

A **retraction** rides the same not-live filter. `<memory>:retract(entry, reason)` withdraws a fact outright — the honest correction when it was filed on the wrong memory, since supersession can only replace an entry *in place* on the same memory, never move it. The projection tombstones the entry by stamping `superseded_by` with the entry's own id, so every live surface hides it exactly as it hides a supersession, and records `retracted_reason` so history shows *why* it was withdrawn. Because relocation would rewrite meaning (visibility resolves per memory), the fix is a two-step: retract here, then re-assert on the right memory. See [Data model → ContentEntry](data-model.md#contententry).

## Search is a third visibility surface

Because private content is embedded and therefore semantically retrievable (see [Storage → Vector store](events-and-storage.md#vector-store)), search is a third way an entry can reach the model, alongside the public-only description and the deterministically-filtered brief, and the predicate governs it identically. It must be held to the same standard: `memory.search` applies `visible(hit, present_set)` to every entry-vector hit before returning it, exactly as brief composition does. A memory's *description* vector needs no filter — descriptions are synthesised from public entries only — but an *entry* vector must pass the predicate before it can surface its memory. Embedding private content is safe only because of this filter; without it, "private hits are tagged as private" would silently degrade into the third-party judgement residual the rest of this document works to bound mechanically.

Surviving private hits carry the inline teller-private marker (`[teller-private, told by … in … (confidential)]`), resolved at retrieval the same way the brief resolves it, not as out-of-band metadata, for the same frozen-context reason. Search is not a back door around visibility; it is the predicate applied to a different candidate set.

A hit carries more than a name. It renders a matched-content snippet so it reads legibly even when the description lags — an FTS5 extract for a lexical match (public content only, so it needs no filter) or the matched entry's clipped text for a semantic entry match (only ever taken from an entry that already passed the predicate). It carries one representative occurrence — the freshest visible dated entry's `occurred_at` over the whole `same_as` class, authored dates outranking extracted — so a recall taken straight off the result line keeps its *when*. And it carries the memory's most salient out-of-class relations (people first, capped at three with a `(+N more)` remainder) so a search for a book club surfaces the cast already on the existing `event/book_club` and the agent recognises it rather than minting a name-guessed duplicate. The snippet and the date are visibility-filtered against the present set — [the participants the platform client reports as currently in the room](conversations-and-briefs.md#present-set-is-per-session-supplied-by-the-client) — like the hit itself; the salient relations mirror the link readers and are visibility-filtered (through `link_visible`) only when an audience is present.

*Implementation note:* because the predicate filters hits after retrieval, search over-fetches beyond the requested `limit` (fetching `max(limit × 4, 20)` candidates) and filters and re-ranks down to it. Candidates are keyed per memory (per [stub](data-model.md#stub-creation-on-first-contact)), so search does **not** collapse two stubs of one merged person into a single hit — both can surface as separate result lines; the class traversal is what dedups *entries* into a hit's occurrence and relations, not what dedups *hits* across a class.

## Direct reads are a fourth visibility surface

The agent also reads a memory *by handle* — `memory.get(name)` then `mem:entries()` / `mem:history()`, and the combined `mem:details` — the read it uses to inspect what it holds about someone before answering. This is a different candidate set from search (one named memory's class, not a semantic query). Treating the by-handle read as an unfiltered window — on the reasoning that the privacy model governs what the agent *surfaces*, not what it may *see* — has a hole the name-conflation case walks straight through. Two unmerged stubs can share a display name; an outsider present under that name asks the agent to recall, the agent reads the same-named stub directly, and relays a confidence the present audience was never cleared for — the subject-guard and teller-gating that search applies are simply absent on the direct path. A marker alone does not close it, because the content is still in front of the model to repeat.

So a direct content read applies the same `visible(...)` predicate, with two deliberate refinements that keep the agent's reach over its own memory intact:

- It **redacts rather than drops.** An entry the present audience may not see stays in the read — addressable (so the agent can still `mem:supersede` it) and still showing its date and teller — but its content is replaced by a stub (`(withheld — a confidence not for the present audience)`). The agent learns *that* a confidence exists, from whom and when, and can decline gracefully ("Dave told me something in confidence — let me check with him"), without ever holding the words it must not relay. This is the read-path analogue of the teller-private marker: legibility where search can afford suppression, because a named read expects *something* back.
- It **fires only with an audience present.** When the present set is empty — a solo compaction flush, a maintenance pass, a wake-up the agent raises to itself — nothing is withheld; the agent sees its whole memory, because there is no one present to leak to and the working-set rebuild depends on it. Redaction is a property of *reading to an audience*, not of the store.

The audience check ignores supersession (it probes with `superseded_by` cleared), so `mem:history()` keeps its purpose — showing the superseded entry — while still withholding it when it is a confidence not for who is present. Direct reads are not a back door around visibility; they are the predicate applied to the agent's own by-handle reach, redacting instead of dropping because the agent asked by name.

The *link* side of a direct read (the link readers `mem:outgoing`/`incoming`/`links`, and `mem:details`) differs in one respect: it **drops** a hidden link rather than redacting it. A link has no text body to stub out, so a link the present audience may not see is simply omitted from the read, and — like the content direct read — the filter fires only when an audience is present. `mem:details` composes both paths, so it inherits redaction on entries and dropping on links.

## Link visibility

Links carry the same audience posture as content entries: a `LinkCreated` event carries `visibility: Visibility` and `told_in: Option<ConversationRef>` (the conversation and turn), mirroring content entries. The read-time predicate `link_visible(...)` mirrors `visible(...)` for links. The subject-guard generalises to the link's target: a `PrivateToTeller` link `A → B` is hidden when B is present, mirroring a `PrivateToTeller` content entry on `person/b` hidden when B is present. For a symmetric relation, both endpoints are subjects — the link is hidden when *either* is present (unless that endpoint is the teller). A link's teller-presence check resolves against its `told_by`, and a link whose `told_by` is absent (a structural or operator-console link with no participant behind it) is treated as **agent-authored** — the `agent` teller is always present, so such a link is never gated by teller-presence.

The predicate is applied at every surface a link can reach the model: the brief's `<relationships>` block, the link readers (`mem:outgoing`/`incoming`/`links`), `mem:details`, and search salient relations. No surface is exempt.

Write-time defaults mirror content: a participant-asserted belief link where the teller is one endpoint and the target is a different person defaults `PrivateToTeller` — "I hate B" is a direct private belief about B. A relayed fact where the teller is neither endpoint (Erin relaying "Dave mentors Grace") defaults `Attributed` — visible to anyone, carrying provenance. Self-links (teller is both endpoints) and non-person targets default `Public`. Agent-authored links about a person must classify explicitly — no default, the same teachable-error gate as content, and it fires only for platform-authority (agent) writes: an operator asserting a link from the console may take the default.

Structural, operator, adjudicated, and `same_as` links are always `Public` (the operator-authored merging `same_as` and the adjudication pass's `same_as` are both emitted `Public`). Inferred links inherit the visibility of the source entry they were extracted from — but link inference reads only a memory's `Public` entries, so in practice an inferred link is always `Public`: the inheritance path exists for correctness (a link extracted from a non-public entry would carry that entry's visibility) but is unreachable while inference consumes only public content.

A surviving non-public link carries the same marker registers as content: none for `Public`, the lighter `[via teller]` for `Attributed`, and the teller-private marker for a confidence — appended to the relationship line (`relation: handle [marker]`), since a link has no text body to prefix.

## The read-time predicate

Teller presence alone is not sufficient. Consider: Erin, alone with the agent, says something private about Marcus, stored on `person/marcus` as `told_by = Erin, PrivateToTeller`. Later Erin and Marcus are both present. A naïve "is the teller present" check passes (Erin is) and the entry would land in Marcus's recent facts in the shared brief, airing Erin's confidence in front of its subject. `PrivateToTeller` encodes private *to* the teller but not private *from* the subject. The predicate adds a subject-guard, and a superseded entry is never live on any surface, so the predicate short-circuits on it first:

```
visible(entry E on memory M, present set P):
  if E.superseded_by is set:  return false   -- a replaced entry is never live
  T       = E.told_by
  subject = subject_participant(M)   -- the participant a person-memory is about; null otherwise
  case E.visibility:
    Public          -> true
    Attributed      -> true   -- visible like Public; the difference is the provenance
                              --   marker it carries and its exclusion from descriptions,
                              --   not what the predicate lets through. Disclosure is judged.
    PrivateToTeller -> teller_present(T, P) AND NOT subject_blocks(subject, P, T)
    Exclude(X)      -> teller_present(T, P)
                       AND no_excludee_present(X, P)
                       AND NOT subject_blocks(subject, P, T)

-- Presence is two-valued, because identity is never inferred: a present
-- participant is either a confirmed same_as-class member of the entity, or not.
presence(entity, P) -> {PRESENT, ABSENT}:
  if some same_as-class member of `entity` is in P:  return PRESENT
  else:                                              return ABSENT

teller_present(T, P):
  if T == agent:      return true    -- the agent is always present to itself
  if T == bootstrap:  return false   -- genesis is never a present participant
  else:               return presence(T, P) == PRESENT

no_excludee_present(X, P): return for all x in X: presence(x, P) == ABSENT

subject_blocks(subject, P, T):
  if subject == null:          return false   -- non-person memory: no subject guard
  if same_class(subject, T):   return false   -- self-disclosure stays visible
  return presence(subject, P) == PRESENT       -- subject in the room -> suppress
```

The implementation returns a *reasoned* verdict rather than a bare boolean — a `VisibilityDecision` with the specific reason (`Public`, `Attributed`, `TellerPresent`, `NotExcluded`, `Superseded`, `TellerAbsent`, `SubjectPresent`, `ExcludeePresent`) — so the console's brief trace can show not just whether a fact surfaced but *why*. The boolean above is that verdict collapsed through `is_visible()`.

Presence resolves through the `(platform, platform_user_id) → memory_id` participant table (a present platform user is resolved-or-minted to a memory id) extended by `same_as` traversal. The traversal follows only *committed* `same_as` edges, and an edge is only ever operator-asserted or adjudicated (see [Cross-platform identity](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated)) — never raw agent inference — so presence stays cleanly two-valued and the predicate has no fail-closed-on-ambiguity cases. A *proposed* merge does not undermine this: a `MergeProposed` is inert (no `same_as`, no `class_id` change), so it never enters the traversal. The "might be the same person" belief exists only as a recorded proposal awaiting adjudication, never as a surfacing state — the predicate sees a merge exactly when, and only when, the edge is committed. The entire ambiguity surface that live inference would introduce still does not exist at the point of surfacing.

`subject_participant(M)` is the participant a memory is about. For a `person/*` memory it is that stub, which the predicate resolves to its class through `class_of`, so "subject present" is class-aware: a private aside about `person/dave@slack` is suppressed when `dave@discord` is in the room, once merged. For every other namespace and for `self`, it is `null`. It resolves by reverse lookup from the memory to its `class_id`, then `presence(class, P)`.

Consequences worth stating:

- *The subject is auto-excluded.* For a `person/*` memory whose subject is present, a teller's private aside about them is suppressed by default, so the agent need not remember to mark it. Self-disclosure stays safe: when `subject` and `teller` share a class the guard doesn't fire, so Marcus's own private statements still surface in front of Marcus.
- *`Exclude` is for third-party carve-outs.* "Everyone except Dave": Erin's aside that also implicates Dave is marked `Exclude({Dave})`. It resolves excludees through the same `presence` machinery as the subject-guard, so `Exclude({dave-discord})` correctly blocks against a co-present `dave-direct` once those stubs are merged. The agent must still name the third party at write time, since only it knows Dave is implicated, so write-time correctness would depend on agent recall; but once named, the read-time block is exact. The write surface is the `exclude` opt on `:append()` and `links.create()`, reconciled before the write records (`reconcile_forced_visibility` in `src/memory/memory_block/mod.rs`): an exclude already fixes the private posture, so a redundant `visibility = "private"` set beside it folds into the exclude rather than erroring — the pairing is the model's reflex for anything sensitive, and the write means exactly what it says — while a `public` or `attributed` beside it is a genuine contradiction and stays a teachable error, as does an `exclude` naming no one.
- *Non-person memories get no automatic guard.* A `topic/*`, an `event/*`, or a free-form `project/*` memory has no participant-subject, so `PrivateToTeller` there is only teller-gated. Excluding a specific party would require `Exclude`. This asymmetry is deliberate: auto-protection is a person-memory convenience, not a universal guarantee.

## Mid-conversation joins re-evaluate the predicate

When someone joins, the new present set — the prior participants plus the joiner — is run through `visible(...)` for the joiner's brief and all subsequent retrieval, so entries transition correctly: a teller joining surfaces their content, and a subject joining suppresses asides about them. The composition mechanics — which brief is rebuilt, how it is injected as a `system` turn, the prefix cache it preserves, and the modelless join path — live in [Conversations and briefs](conversations-and-briefs.md#mid-conversation-joins). Content already emitted into pre-join context can't be retracted; for that material the compartmentalisation principle (below) is the backstop. But the dangerous direction is fully closed, because the join-brief runs the predicate over the present set that includes the joiner.

## Synthesis traverses the `same_as` class

Description regeneration and belief arbitration read the whole class's content — the graph's `class_entries(id)` traversal, which unions the live entries of every stub in the `same_as` class (composed over the per-stub `entries_local` primitive) — producing one unified per-class result. There is no per-stub-description case: `same_as` means the same human, since the operator only ever merges identical humans, so a multi-stub class is one person who should have one self-description, and a Discord-description drifting from a direct-interface-description would be wrong. Genuinely distinct people are simply never merged, so they stay separate classes with separate descriptions for free.

The two synthesis consumers read *different* visibility slices of that class-wide content, and this is load-bearing:

- **Description regeneration** consumes only the class's `Public` entries. This is what guarantees "nothing teller-private (or attributed) ever bakes into a description" — the public-only description rule is enforced by the input filter, not by trust. An `Attributed` entry, though visible to anyone at read time, is deliberately excluded here so its provenance marker is never laundered off into markerless prose.
- **Belief arbitration** (the contradiction check that flags competing entries) reads the wider `Public` + `Attributed` slice, so two relayed-but-conflicting secondhand accounts the agent marked `Attributed` are still arbitrated against each other.

## Defaults at write time

The `PrivateToTeller` default exists to guard asides about an absent person; it is not a general default, but a *floor* the agent classifies up from rather than a verdict. An explicit `visibility` choice on the write is always honoured verbatim. Absent one, the resolution order is: a `#confidential` room firms everything to `PrivateToTeller` first (see below); otherwise the subject/namespace default applies.

- A `person/*` memory where a **participant** relays something about someone else (`subject` non-null, `told_by` a participant ≠ `subject`, compared at the write-time stub, not the class) is the classification case (see [Visibility versus disclosure, and three postures](#visibility-versus-disclosure-and-three-postures)). The agent reads the framing and chooses: `private` (`PrivateToTeller`) for a genuine secret, `attributed` for an ordinary secondhand fact, `public` for something plainly open. Left unclassified, it defaults `PrivateToTeller` — the fail-closed floor, so a missed call errs toward privacy. The default is never `Public`: an unmarked confidence defaulting shareable is the leak this guards. The agent's job is to *upgrade* the everyday majority to `attributed`, so remembering Dave's role does not require Marcus in the room.
- A `person/*` memory written by the **agent itself** (a synthesis or pre-compaction-flush re-recording, `told_by == agent`) has **no default** — it must classify its visibility explicitly, or the write is rejected with a teachable `VisibilityRequired` error. The participant-aside mechanism keys on a participant teller, so an agent-authored entry can't inherit it; silently defaulting it `Public` is exactly how a re-recorded confidence leaks. This forces the judgement the agent already understands rather than relying on it to reach for an optional flag (a gating flush-visibility oracle guards the seam).
- Self-disclosure (`told_by == subject`, by stub) → `Public`.
- Every non-person memory — the other recognised namespaces (`place/*`, `event/*`, `topic/*`, `context/*`), `self`, and any free-form name (`project/*`, `concept/*`, or a bare handle, none of which is a recognised kind), all of which have `subject == null` → `Public` (including agent-authored content: only a `person/*` subject triggers the required-classification rule above).

The **confidential-context override** sits ahead of all of these: when the conversation's `context/*` room carries the `#confidential` tag, every unclassified write — content or link, person or non-person — firms to `PrivateToTeller`. So in a room marked confidential even a non-person memory (a `topic/*`, or a free-form `project/*` note) defaults private, without the agent having to reach for the flag. An explicit choice still wins over the override.

Sensitivity inference (below) is the upgrade path for the ordinary case: a non-person memory whose content is actually sensitive — a confidential project, say — gets bumped, rather than everything defaulting closed. Defaulting non-person memories to `PrivateToTeller` unconditionally would silently fragment project, topic, and event knowledge by teller-presence: the agent could not discuss the Hooli project unless whoever mentioned it were in the room. That is over-suppression, not safety, so the default for things-and-rooms is open and the agent (or a confidential room) tightens deliberately.

## The `agent` teller

Content the agent authors itself — an observation it forms, an inference it draws, an `Initiated` wake-up it records having raised, a self-disclosure about itself — is recorded with a reserved `agent` pseudo-teller, distinct from the `bootstrap` genesis source. `agent` is defined as always present to itself, so `teller_present(agent, P)` is always true and agent-authored entries pass the teller-presence check like any teller's own statements; `bootstrap`, by contrast, is never a present participant (its content is `Public`, so this never gates it). Self-disclosure and non-person content default `Public`; agent-authored content **about a person** is the exception — it has no default and must set its visibility explicitly (see [Defaults at write time](#defaults-at-write-time)), because the participant-aside mechanism can't fire on the agent teller and a re-recorded confidence silently defaulting public is a leak. The subject-guard still applies to whatever visibility it carries if the note is about a present person. This gives agent-derived memory a coherent provenance instead of an undefined or borrowed teller.

## What `PrivateToTeller` actually promises

It surfaces whenever the teller is present, never to the subject, and to other co-present third parties only as a flagged judgement call. It does not mean "stays with the exact audience it was said to": if Erin tells the agent something about Marcus while alone, and later Erin and Dave are present (Marcus absent), the mechanism permits that entry to surface to Dave.

Teller-gating is chosen over audience-gating deliberately. Binding each entry to the participant set present when it was recorded would over-suppress, making the agent useless at its core job of building a picture of someone from what others say. The price is that the residual third-party case is governed by agent judgement, not by mechanism. `Exclude` is the lever for when the agent knows a specific third party should be carved out.

Teller-gating is the right behaviour precisely *because* the entry is a confidence — a secret should resurface only when its owner is around to vouch for it. Ordinary facts ride `Attributed`, which the predicate passes regardless of who is present, so teller-gating is reserved for confidences and never over-suppresses the common case: Dave's role does not vanish the moment Marcus leaves the room. `Attributed` carries the provenance marker — it still says "Marcus told you this" — without binding the fact's *visibility* to Marcus's presence; the binding remains only where the content actually warrants it.

## Sensitivity inference

The agent should bump visibility toward more private on signals like topic class (health, finances, relationships, work struggles), hushed register or explicit markers ("between us," "don't tell"), asymmetric context (talking about someone in their absence), and the confidentiality of the current conversational context (a private channel or DM raises the default — see [Conversations and contexts](conversations-and-briefs.md#conversations-and-contexts); a `#confidential` room already firms unclassified writes private mechanically). When uncertain, the agent asks before writing: *"That sounds personal — should I keep this between us, or is it okay if it comes up later?"* One question now beats an incident later. This is a model-judgement call and is exactly the kind of thing the validation scenarios must exercise (see [Validation](observability-and-testing.md#validation-and-the-eval-harness)).

## Provenance markers, and compartmentalisation

Even with a filtered brief, the agent can leak by inference, and the residual third-party case is delegated to its judgement by design. So a relayed fact reaches the agent flagged with its provenance rather than presented as neutral fact, in one of two registers matching its posture:

- A surviving `PrivateToTeller` or `Exclude` entry carries the **teller-private marker**: who told it, that it was private, the room it was told in (`told_in`) and — when that room is `#confidential` — that it was said in confidence, plus a `[turn:<ulid>]` token for cross-linking — `[teller-private, told by Erin in #leads (confidential) [turn:01KX…]]`. This is a confidence reaching the model because its teller is present; the marker tells the agent to treat it as a judgement call, not neutral fact.
- An `Attributed` entry carries a lighter **provenance marker** — `[via Marcus [turn:01KX…]]`, or `[via Marcus in #general [turn:01KX…]]` — naming the source without the language of confidence. It reaches the model unconditionally (the predicate always lets `Attributed` through), so the marker is the whole signal: this is secondhand, weigh it as such before volunteering it, and remember that a casual relay is not the subject's own account.

Both are built the same way (resolving `told_by` to a display name and `told_in` to its `context/*` room at build time) and differ only in wording, so the agent reads the posture off the marker. The room's display name is its handle with the namespace stripped and a `#` prefixed (`context/leads` → `#leads`), and the `#confidential` flag is read from that room's tags — a deterministic lookup baked into the rendered marker.

The marker must render inline in the text the model sees — for example `[teller-private, told by Erin in #leads (confidential) [turn:01KX…]]` — not as out-of-band metadata, because the brief's recent facts are plain text frozen into the system prompt. If the marker were attached only at retrieval time, frozen pre-join facts would carry no marker, and the cross-context judgement (this was said in a confidential room, so be careful repeating it elsewhere) would have nothing to act on. The brief composer resolves `told_in` to its `context/*` memory and checks the `#confidential` tag at build time and bakes the result into the rendered marker. The `[turn:<ulid>]` token is embedded (inside the marker's closing bracket) so the frontend's `scanTurnRefs` can render a cross-linkable chip.

The system-prompt principle: *an entry marked teller-private was told to you in confidence; the subject will never see it and named excludees are already filtered, but surfacing it to any other co-present third party is a judgement call — stronger still if it was told in a room marked confidential. When unsure, hold it or check with the teller.* If answering would require revealing such context, the agent flags rather than answers: *"I have some context from elsewhere that might be relevant — let me check with [teller] before bringing it in."*

## Defence in depth, with a clear boundary

The mechanism enforces what can be stated as an invariant: the subject never sees a private aside about them, named excludees are honoured absolutely, and presence is exact because identity is never guessed. Agent judgement handles the irreducibly contextual third-party residual, informed by the marker.

The failure is bounded in system state but not in the world. A judgement lapse leaves no durable artefact — nothing teller-private (or attributed) ever bakes into a description, which the public-only description rule guarantees, and nothing replays into prose — but an aired confidence cannot be retracted from the third party's memory. The marker exists precisely because that consequence is real even though its system footprint is bounded. The confidence tier stays permissive on the co-present third-party case because over-suppression there costs real usefulness; ordinary facts ride `Attributed` and never depend on this judgement, so the residual delegated to the agent is the confidence tier's alone.

## The acceptable failure mode

Default-private means the agent sometimes has less to say about a participant than it "should." That's the right trade. The system prompt acknowledges it: *"I know less than you might expect because most of what I've heard about you came from others, and I'm keeping that to itself"* is a fine thing for the agent to say.
