# Time

Four distinct concerns: when something happened, when the agent learned it, what "now" means, and what's expected to happen.

## Bi-temporal entries

`asserted_at` (when the agent recorded it) and `occurred_at` (when it happened) are the two timestamps on every entry (see [Data model](data-model.md)); `asserted_at` is always present; `occurred_at` is optional and possibly vague. Bi-temporal agent memory is not novel here: Zep and Graphiti are the closest prior art on this axis and the inspiration for it. The model leans further on the *occurred* side, with `BeforeAfter` anchored to other memories, `Approx`, and `Recurring`.

## TemporalRef

A small typed vocabulary, not free-form strings:

```
TemporalRef =
  | Instant(timestamp)                 -- "2025-03-14T09:30"
  | Day(civil-date)                    -- "2025-03-14"
  | Range { start, end }               -- "March 2025", "Q2 2024"
  | Approx { center, fuzz_days }       -- "around 2019"
  | Recurring(rrule)                   -- "every Tuesday"
  | BeforeAfter { dir, anchor }        -- "after Dave's wedding"
```

The agent writes it as a tagged Lua table whose single key names the variant: `{ instant = <ms> }`, `{ day = "YYYY-MM-DD" }`, `{ range = { start = <ms>, ["end"] = <ms> } }` (`end` is a Lua keyword, so in Lua source the key is quote-bracketed), `{ approx = { center = <ms>, fuzz_days = <n> } }`, `{ recurring = "<rrule>" }`, or `{ before_after = { dir = "before" | "after", anchor = "event/..." } }`. `Approx`'s fuzziness is a symmetric tolerance in whole days (`fuzz_days`), not a free-form span.

The agent picks the most specific type it can justify. A `BeforeAfter` anchor is another memory (for instance, `event/dave-wedding`), forming a temporal graph alongside the relationship graph. The type is dependency-free and pure — it borrows the parent time module's civil-date math and never reaches into the graph itself; anchor resolution, the one graph-dependent step, is done by the materialiser, which passes the resolved anchor's bounds *into* the pure `bounds` derivation.

A small extraction pass turns natural-language phrases ("last Tuesday," "before the move") into structured refs, running in the same turn-end model call as description regeneration (the [`synthesize` pass](write-path.md#coalesce-then-regenerate-once)). The call's schema asks for one occurrence per statement that names a real-world time; because a language model cannot compute epoch milliseconds, the model emits ISO date strings (`"2026-06-03"`, occasionally a datetime), which are mapped to the stored `TemporalRef`. The statements the pass is shown carry their metadata in a leading bracket — teller, assertion date, and, when the entry already has one, its recorded occurrence (`· occurred 2026-10-03`) — so a phrase in one statement that points back at another's date can anchor to that stated occurrence rather than to the conversation's now. The mapping and its scope follow a few rules:

- **Leniency.** A bare `YYYY-MM-DD` given under the `instant` tag is taken as a `Day` (a live probe showed the model uses the two interchangeably).
- **Rejection.** A `Recurring` rule the build cannot interpret — a free-phrasing such as "every Monday" rather than an rrule — is rejected rather than committed as an entry that would parse to no occurrence and so silently never fire.
- **Drop, not discard.** A statement whose extracted time will not parse at all is dropped with an `EntryTemporalResolveFailed` event (logged, not fatal), so a malformed occurrence never discards the good description alongside it.
- **Back-pointing phrases resolve against siblings, not now.** The extraction instructions resolve a phrase that points back at a date given elsewhere ("this date," "that day") against the stated occurrence of the sibling statement it refers to — visible in that statement's bracket — and omit an occurrence entirely when no time and no sibling anchor it, rather than defaulting to the current day. A fabricated now-relative date reads back as fact, so it is worse than no date.
- **Another referent's date is not the subject's.** A statement can mention a genuine, pinnable date that describes a *different* referent than the statement's subject — an inspiration, a namesake, a comparison, or a historical analogy ("X is named after Y, from 1902"). The instructions leave such a statement unextracted: the date belongs to the other referent, and the subject's own fact, being timeless or present, gets no occurrence rather than someone else's date.
- **Current-day guard.** Because the model can still slip and anchor such a phrase to "Current time," the application path double-checks: when an extracted occurrence lands on the current day *and* a sibling entry of the same memory (the description mirror among them) carries an occurrence on a different day, the resolution is suppressed — recorded as an `EntryTemporalResolveFailed` naming the suppression — and the entry stays untimed rather than contradicting the authored date. The check is conservative: only the single-day shapes (`Day`, `Instant`) can trip it, and only against a differently-dated single-day sibling, so a genuinely same-day fact or a resolution on any other day applies normally.
- **Scope.** The extraction only ever touches *untimed* entries written this turn: an entry the agent already stamped with an `occurred_at` at append is left alone, and a description-mirror seed entry is flagged and skipped so its untimed prose never acquires a fabricated occurrence that would collide with a later dated append.
- **Private entries.** A focused variant of the same pass resolves an instance's *private* entries separately, so each pass only resolves the entries it was actually shown.

*Watch-list handled:* a `BeforeAfter` anchor can point at a memory later soft-deleted. Anchor resolution is deliberately **not** filtered by soft delete — `MemoryDeleted` preserves contents, so a deleted anchor's occurrence stays resolvable rather than silently collapsing to empty bounds. The anchor's bounds are resolved from the *projection so far*, taking the anchor's earliest timed entry (`ORDER BY occurred_sort LIMIT 1`); an anchor first timed *after* the `BeforeAfter` entry was appended therefore does not retroactively re-resolve it — resolution is a one-time denormalisation at materialisation, not a live join.

## Storage and resolution

The typed value is stored as tagged JSON in `occurred_at`, plus three denormalised columns computed at materialisation for ranking and calendar queries: `occurred_sort` (one representative instant) and `occurred_lo` / `occurred_hi` (a bounding interval). The derivation (`TemporalRef::bounds`) is pure for every variant except `BeforeAfter`. Per variant:

- `Instant(at)` → sort = lo = hi = the instant.
- `Day(date)` → sort = noon (midnight + half a day), lo = midnight UTC, hi = the day's last millisecond (midnight + a day − 1). An unparseable or impossible civil date (`2026-02-29` in a non-leap year) yields **empty bounds** rather than rolling over into March.
- `Range { start, end }` → sort = the midpoint, lo/hi = the ends, with a reversed range normalised so lo ≤ hi rather than producing a negative interval.
- `Approx { center, fuzz_days }` → sort = centre, lo/hi = centre ± (`fuzz_days` × a day).
- `Recurring(rrule)` → no fixed instant: sort/lo/hi are all null. Concrete instances are computed on the fly by the recurrence expander (`next_occurrence`) for wake-up arming and calendar expansion, never expanded into the log.
- `BeforeAfter { dir, anchor }` → resolve the anchor's representative bounds (its earliest timed entry, read from the projection so far, unfiltered by soft delete), shift the sort by a nominal one-hour epsilon (`BEFORE_AFTER_EPSILON_MILLIS`) in `dir`, and propagate the anchor's interval when it is vague (lo ≠ hi), shifted with it; for a point anchor the shifted instant is the whole interval. An unknown or itself-untimed anchor yields empty bounds — a vague or missing anchor is agent input, not a failure.

A fourth denormalised flag, `occurred_authored`, records whether the agent stamped the occurrence at append (`1`) or the extraction pass resolved a previously untimed entry later (`0`; an `EntryTemporalResolved` event always sets it to `0`). Every surface that renders one representative date for a memory — a search hit, a link's neighbourhood line — prefers the freshest *authored* occurrence over any *extracted* one, so a clock-anchored guess never shadows a date the agent actually stated. An extracted date surfaces only when the memory's [`same_as` class](data-model.md#cross-platform-identity-operator-asserted) holds no authored date at all.

## "Now"

Declared in the system prompt at conversation start, since the model shouldn't infer it from training data. Beyond that anchor, each inbound and injected turn in the buffer is prefixed with the wall-clock time it was recorded, derived from its `recorded_at` — so "now" stays current without a drift heuristic or any rewrite of the frozen prompt. The stamp is a `[<time>] speaker: text` (or `[<time>] text` for a system turn) prefix rendered at buffer-assembly time. The agent's own turns are left unstamped: its replies are rendered back to it as history, and a stamp there would teach it to emit timestamps into its replies (a time it can't actually know), so only the messages it *reads* carry one. The per-turn stamps live in the buffer suffix and cost no extra events: each is read off the turn's existing `recorded_at` and frozen the moment the turn lands, so it stays part of the cacheable prefix (only the live turn's stamp is uncached) and replays deterministically. They give the agent a timeline a single session-start time can't: the gap between two messages, how long a participant was quiet.

## Calendar as a view over memory

No separate calendar store. Future events are memories with a future `occurred_at`:

```
event/dentist-2026-06-03
  description: "Dentist appointment, 9am"
  contents: [{ asserted_at: 2026-05-20, occurred_at: Instant(2026-06-03T09:00),
               text: "Scheduled cleaning", told_by: marcus }]
  tags: [#scheduled, #health]
```

The calendar surface is queries over memory, all reads; each read marks its result memories as touched, updating their last-access recency:

- `calendar.upcoming({ within = "7 days" })` — memories with a concrete occurrence within the window of now, soonest first (default 7 days).
- `calendar.overdue({ within = "14 days" })` — memories whose dated occurrence has already *passed*, looking back over the window (default 14 days), soonest first. This is what a "what should I be on top of?" check needs that `upcoming`/`on` miss: they look at today and ahead, so a reminder whose day slipped is otherwise invisible. Recurring occurrences are deliberately excluded from `overdue` — a recurring entry's next instance is always ahead, so it is never "overdue"; its comes-due nudging is the wake-up scheduler's job.
- `calendar.on("2026-06-03")` — memories with an occurrence on a given civil day. Accepts a date object or a `"YYYY-MM-DD"` string.
- `calendar.recurring()` — a listing of memories that carry a recurring occurrence; instances are not expanded.

`upcoming` and `on` merge concrete occurrences with the **next in-window instance** of each recurring entry (computed virtually via `next_occurrence`, anchored at the entry's `asserted_at`), ordered by instant, so a weekly standup interleaves with one-offs; `overdue` uses concrete occurrences only. The console's calendar view expands a recurrence into *every* instance in a window (up to a cap per entry) through the same expander, so agent scheduling and the rendered calendar cannot drift.

Anything can be calendared: "Marcus said he'd review the spec by Friday" becomes a memory with `occurred_at: Day(friday)` and `#due`. A deadline mildly overloads `occurred_at`, which we accept deliberately rather than duplicate the temporal machinery with a parallel `due_at` field. A plan whose milestones fall on different dates is several dated facts, not one: each milestone rides its own dated entry (or event memory) under its own `occurred_at`, so each date stays independently addressable and calendarable rather than bundled into a single entry stamped with the first. The brief includes a small [`<upcoming/>` block](conversations-and-briefs.md#composition) (a 7-day window, capped at 5 items by default, each filtered through the [visibility predicate](visibility.md)) so the agent organically raises near-future items.

## Date arithmetic is the runtime's job, not the model's

Resolving "this Friday" to a calendar date is arithmetic, and a language model does arithmetic unreliably — even given the current weekday it slips by a day often enough to mis-schedule a one-off reminder. The fix is to stop asking it to: rather than have the agent compute a date in its head and write the result as a string (`occurred_at = { day = "2026-06-12" }`, sometimes off by one), the `calendar` global hands it **typed date objects and the operations to derive them**, so the model's job drops from *computing the date* to *naming the operation* — the easier, more reliable task, and one the runtime then executes deterministically.

The constructors are `calendar.today()`, `calendar.next("friday")` (the soonest date on or after today with that weekday — `today()` itself when today matches), `calendar.in_days(n)` / `calendar.in_weeks(n)` (relative to today, negative going back), and `calendar.date("2026-06-12")` (parse). A date object carries calendar-correct arithmetic — `:add_days(n)`, `:add_weeks(n)`, `:add_months(n)` (so 31 Jan + 1 month is 28/29 Feb, not 3 March), `:weekday()` (its weekday name) — plus `:to_string()`. It prints and concatenates as its ISO day, so a backtick-interpolated `` `Reminder for {calendar.today()}` `` stringifies without an explicit call. A date object **is** a valid `occurred_at`: it is a `{ day = "…" }` table backed by a metatable, so `ev:append("Send the board update", { occurred_at = calendar.next("friday") })` flows straight into a `Day` `TemporalRef` with no intermediate string the agent could miscompute. "The Friday after next" is `calendar.next("friday"):add_weeks(1)` — composed from operations, never arithmetic carried in the model's head. The same normalisation lets a date object (or a bare date string) stand in wherever a millisecond timestamp is wanted inside a tagged `occurred_at` table — a `range` endpoint or an `instant` — coercing it to the day's bounding instant (a range from Monday to Friday spans all of Friday).

This is day-precision (a civil day, the common case for agent scheduling); a precise time of day still rides the explicit `{ instant = <ms> }` form. It complements rather than replaces the extraction pass: extraction resolves a relative phrase left in an entry's *text* after the fact, while these operations let the agent compute the date *up front* — the scaffold steers it to the latter, so a date is correct at write time rather than relying on a later pass to repair it.

## Scheduled work

Wake-ups surface into sessions — the bounded activity windows defined under [Conversations and briefs](conversations-and-briefs.md#durable-conversation-vs-session) — at session open. A scheduler derives wake-ups from events: the calendared memories are the pending triggers, and when one comes due it emits `ScheduledJobFired`, which the orchestration layer turns into action. A **concrete** trigger comes due when an entry's representative instant (`occurred_sort`) passes `now` and was later than its own `asserted_at` — scheduled for the future when recorded, not a past event logged after the fact. So a dentist booked for next week fires; "the dentist was last week," recorded today, never does, and the surface doesn't fill with historical `occurred_at`s. A **recurring** trigger's next instance is computed strictly after its last firing (`fired_at`, or `asserted_at` if it has never fired) and is due when that instance is at or before `now`; each firing re-arms it, so exactly one trigger is live per recurring entry and a long-idle agent fires one catch-up per entry rather than a backlog. The firing is recorded in the log rather than recomputed from a live clock, so the surface is a function of the log and replay reconstructs both the calendared memories and the fired state. The mechanism is generic over job kinds; event-attached wake-ups are the species in use ("on the morning of `event/dentist...`, surface it to Marcus"). The same mechanism handles past-anchored wake-ups (anniversaries, "it's been a year since Marcus mentioned X").

The firing mechanism is the `fire_due` operation behind `ScheduledJobFired`, with the drained surface below. `fire_due` runs in two places with identical, global semantics — it fires every due trigger across all conversations, not just the opening one: a background driver (`Instance::run_scheduler`) runs it continuously on a `tokio::interval` (default every 60 seconds, `scheduler.tick_seconds`), so a long-idle agent's reminders fire on time rather than waiting for a conversation; and the session-open path runs it as a catch-up, so a just-due item is caught the moment a conversation resumes. A fire failure in the background driver is logged, never propagated — the driver is long-lived and must outlast a transient store or graph error. The eligible subset is then *surfaced* per session by the open-time drain. Both run on the long-running shared-server host (see [Concurrency](write-path.md#concurrency)).

## Agent-initiated speech

A fired wake-up wants the agent to say something unprompted, but deciding when unprompted contact is welcome is hard and out of scope here. The compromise: the turn schema distinguishes `Initiated` from `Responding` (an `Initiation` enum on the conversation turn), and a fired wake-up sits in a *computed surface* — entries that have fired (`ScheduledJobFired`) without being surfaced, tracked by a lightweight `surfaced_at` marker. The agent never pushes. The surface is drained at the start of the next eligible session, where eligible means both that the item is `visible(...)`-permitted against the present set and that the item targets a participant who is present. The target is the memory's subject — a `person/*` memory resolves to that person — and the entry's teller if a participant; an agent-authored item on a non-person memory has no target and isn't raised proactively. A dentist reminder told by Marcus, private, targeted at Marcus is thus not drained into a stranger's session. The drain is capped at `scheduler.max_wakeups_per_session` items (default 5). Draining records `ScheduledItemSurfaced`, setting `surfaced_at` so the item isn't raised again, and injects the content as an `Initiated` system turn the agent sees in its buffer. The injected turn names the due items as reminders to bring to the present participant, framed as a directive to relay them in the agent's own words — not a bare data block, because an unexplained list reads as internal scratch state the model does not reliably surface; the framing even tells the agent to treat every listed item as due now, so it does not dismiss a recurring reminder because its weekday is not today. This delivers the useful 80% — the agent appears to remember and raise things for the right person — without solving the interrupt-a-human problem. (Besides drained wake-ups, the only other producer of an `Initiated` turn is the pre-compaction / checkpoint flush, which acts unprompted over unflushed working state.)

The surface drains at **session open**: a wake-up that comes due while a session is already live is recorded on time by the background driver and surfaces at the next session boundary.

## Recency and volatility

The recency boost in search uses `occurred_at` when present, falling back to `asserted_at`, so an entry written today about 2019 retrieves like a 2019 memory by relevance. `volatility` modulates the boost: high-volatility facts (employer, location, current project) decay sharply, low-volatility ones (birthplace, date of birth) barely. The boost is roughly `decay(now − relevant_time, volatility)`. Briefs render times relatively ("last week," "Wednesday") at build time.

Volatility is the agent's to set — `<memory>:set_volatility("high" | "medium" | "low")`, or inline on an append (`volatility = "high"`) so a fast-changing fact is classified as it is recorded rather than in a separate call. The cue is the nature of the fact, not its wording: a current role, project, location, or status is `High`; a name or a birthplace is `Low`. The one field drives two distinct mechanisms. The first is the *ranking* decay above: gentle, continuous, applied to every memory. 

The second is **staleness**, a binary legibility signal in the [marker family](visibility.md#provenance-markers-and-compartmentalisation) (`[disputed]`, `[via …]`). An entry on a `High`-volatility memory that has aged past a short horizon (30 days), *and that nothing newer has replaced*, reads with a `[stale — no newer entry]` marker on every live surface (brief, search, direct read). So the agent surfaces it as possibly out of date — "last I heard …," or an offer to confirm — rather than asserting a fading fact as current. The marker only ever rides an unreplaced entry: a live surface excludes superseded entries, and `mem:history()` — the one place a superseded entry still shows — suppresses it, so the "no newer entry" claim always holds and the marker signals a fact to hedge or reconfirm, never one with a newer version sitting somewhere to hunt down. Staleness is **opt-in and `High`-only**: `Medium` (the default) and `Low` never go stale, so the marker is never a false alarm on a durable fact, and the agent must deliberately mark a memory fast-changing for any of its facts to read stale. It is age-based, not audience-based — a fact is stale regardless of who is present — and the 30-day horizon sits well inside the `High` ranking `τ` (90 days), so a fact reads stale (prompting a hedge) while still ranking as relevant enough to surface.

## Search scoring (starting defaults)

Relevance combines semantic similarity, lexical match, and tag overlap — the default weighting is `0.5·cosine + 0.3·bm25_norm + 0.2·tag_match` — then adds a bounded recency bonus of up to `+0.3` of the form `exp(−Δt / τ(volatility))`, with `τ = 90 / 365 / 3650` days for High / Medium / Low volatility. `Δt` is measured against the memory's most recent relevant time over its whole `same_as` class — each entry's `occurred_sort` when present, else its `asserted_at`, falling back to the memory's creation time — clamped so a future-dated occurrence (a calendar item) counts as no decay. Recency informs ranking without dominating it. These constants are tuning knobs (the `SearchSettings` / `RecencySettings` block), set concretely and overridable via config.

## Sequence vs wall-clock

The log's `seq` is the primary timeline; wall-clock timestamps are a denormalised convenience for human-readable queries and recency math, with `seq` breaking ties. "What changed since snapshot N" is a `seq` range; "what happened Tuesday" is a wall-clock range. They are not interchangeable. The clock itself is an injected seam (`Clock`): production reads the system UTC clock, while tests drive a `ManualClock` so recency decay, calendar windows, and scheduled wake-ups are exercised without real time passing.
