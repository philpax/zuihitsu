# Trust and authority

## Trust model

The trust model rests on two postures. The mechanisms these postures rest on — the visibility machinery, the `self` memory, `same_as` merges — are each defined in the area documents that follow; this document states who is trusted with what.

### The operator is trusted

A single operator runs the instance, owns the event log and the binary, holds console access, and creates or destroys the agent. Mechanisms keyed off this trust are appropriate: imprinting the creator through a console-launched interview, console access to internal state, the ability to assert that two platform-identities are the same human, and the ability to inspect the event log. The operator runs on hardware they control and is not an adversary to the system.

### Participants are not trusted with each other

The agent meets many people. They tell it things about themselves and about each other, and they have legitimate competing interests against one another even though none of them is attacking the system itself. Four participant-facing channels need bounding even in the trusted-operator deployment:

- *Confidences between participants.* The [visibility machinery](visibility.md) (per-entry tellers, the subject-guard, and the teller-private marker) keeps one participant's asides about another from reaching their subject. This is the core correctness concern of the whole system.
- *Participant-induced network I/O.* The agent's only outward reach is the operator-configured MCP servers (see [Lua API → External I/O via MCP](agent-loop.md#external-io-via-mcp)), and the fetch itself happens in the MCP server, so the SSRF/egress surface lives there rather than in the agent. A participant who can steer the agent into fetching a URL is therefore a real surface. We accept it under the trusted-operator, loopback-only assumption and catalogue it in [Known limitations](limitations.md) with the egress hardening it needs before untrusted exposure.
- *Self-model writes.* The agent's own [`self`](data-model.md#naming-conventions) memory is writable only under operator authority, never from an ordinary platform conversation. A participant cannot durably reshape the agent's self-model by asserting "you're really just a support bot," because that write has no path to `self`. The agent can still learn about itself from anyone; those observations land on the relevant [`person/*`](data-model.md#naming-conventions) memory or as teller-marked content that does not enter `self`'s own disposition.
- *Retroactive mutation of another participant's records.* What one participant entrusted to the agent must not be quietly rewritable from another participant's conversation. Two mutations with retroactive or suppressive reach are structurally gated. Removing a `#confidential` tag is operator-only. The teller-private marker resolves a room's confidentiality at read time, so removing the tag weakens the disclosure-judgement signal on every historical aside told under it — a broadcast, retroactive change, the same shape as a `self` write. The gate is asymmetric: the agent may still *add* the tag from conversational cues, because over-marking merely hedges. Separately, a platform turn may not supersede a non-public entry told by a participant outside the current speaker's [`same_as`](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated) class. Superseding drops the entry from every live surface, so the correction path from someone else's turn is an appended entry, never suppression of what another participant confided. Volatility changes, by contrast, remain agent judgement: the staleness marker is a hedge, not a gate (see [Known limitations](limitations.md)).

### The operator has no platform identity and no conversational privilege

"Operator" is not a participant the agent can recognise in a chat; it is whoever holds the console. Operator-level acts — creating the agent, asserting a `same_as` merge, editing `self`, and running the imprint interview — all happen through the console, which connects directly to the agent server and authors its writes as `source: Operator`. In ordinary conversation the operator is just another participant with an ordinary `person/*` memory and no special powers. This collapses the self-model-injection concern to a single invariant — nothing reachable from a platform conversation can write `self` — and removes any need to authenticate an "operator principal" on a per-turn basis.

The operator's own facts live on their real `person/<name>` profile, which a `same_as` merge folds into a dedicated `person/operator` anchor. The anchor is a pure merge target: a content write to it is rejected under *any* authority (only its merge and origin links are permitted), so operator facts always live on the merged profile rather than accreting on the anchor.

### Out of scope

Adversarial operators, prompt injection through externally-fetched content (real once the agent fetches via MCP — see [Known limitations](limitations.md)), side-channel attacks on the local model, supply-chain attacks on the binary, and open-internet exposure to untrusted participants. These become real the moment the deployment context changes: shared hosting, multi-tenant, or untrusted operators. The corresponding hardening is catalogd in [Known limitations](limitations.md) and gated on that transition.

## Clients and the server boundary

The **agent server** is the only process that touches the event log, the materialised graph, and the model. It exposes a single API — the structured surface the console already shares (see [Observability](observability-and-testing.md#observability)) — and every other actor reaches it as a client. Authority is a property of the client's role, enforced server-side, never a property of where the client runs or what a participant types.

The binary's default invocation *is* that server: running `zuihitsu` with no subcommand boots the long-running instance, which binds a loopback HTTP port by default (`127.0.0.1:7777`, overridable in the `[serving]` config), serves the API, and runs the background scheduler alongside the indexer, describer, adjudicator, link-inference, idle-sweep, checkpoint-sweep, and snapshot workers. The operator CLI subcommands and the web console are clients of the running instance, not separate openers of the store — only one process holds the single-writer log lock (see [Boot](lifecycle.md#boot-every-startup)), so the CLI reaches the agent *through* the server rather than around it. (Two CLI subcommands are the exception, opening the event log directly because the running server takes no lock on it: `events` reads it read-only, and `revert` truncates it and so requires the server stopped.)

The served surface is split into three groups, mirroring the two authority roles (platform and control) plus the console. The operator route group lives under `/control`, the participant route group under `/platform`, and everything else falls through to the embedded web console. Each authority group carries its own auth layer as a per-sub-router middleware rather than one top-level layer, so a control key never authorises `/platform` and a platform key never authorises `/control`. The console is compiled into the binary (`rust_embed`, from `console/dist-embedded`) and served at the root: an asset by path, or `index.html` for any client-side route so a deep link or refresh lands in the single-page app. Because it is served from the agent's own origin, it connects back to `/control` keylessly as a loopback peer, and its HTML shell boots in `agent` mode.

Two authority roles:

- **Platform clients** — the web console's direct interface and the CLI's `send`/`join`. They deliver participant turns (`POST /platform/message`) and note mid-session arrivals (`POST /platform/join`), and carry no operator authority: they can act only as the participants they represent. Each stamps every turn with its `(platform, platform_user_id)` in the request payload so the server resolves it to a stub. They cannot reach the operator-only endpoints, which is what makes "the operator has no platform identity" (see [Trust model](#trust-model)) actually enforceable.
- **Control clients** — the console and CLI. They carry operator authority: creating the agent, asserting a `same_as` merge, running the imprint interview, and editing `self` (there is no dedicated `self` endpoint — the operator edits it by running Lua through `POST /control/lua` or by conducting the imprint interview, both of which execute under `Authority::Operator`). Their writes land as `source: Operator`. Operator authority is granted by reaching the control surface as a trusted peer: a loopback peer is trusted without a credential (the local CLI and console), while a remote peer must present a valid per-surface API key (`Authorization: Bearer <key>`, see [Known limitations](limitations.md)). Presented keys are compared by fixed-width SHA-256 digests, scanning the whole configured list unconditionally, so neither key length, shared prefixes, key count, nor match position leaks through timing. This is fail-closed — with no keys configured, only loopback holds operator authority — so binding a routable interface does not expose an unauthenticated surface: a remote peer without a key is rejected with `401`. (A reverse proxy would make every peer appear loopback, so the surface must not be fronted by one without re-checking auth.)

**Transport and authority are orthogonal.** A Discord adapter compiled into the server binary is still a client; it just uses an in-process transport instead of authenticated HTTP, and it still holds only platform authority. An in-process adapter, a remote HTTP adapter, and an in-memory test harness driving the server are the same client at different transports. The one hard rule: co-location is never an authority escalation and never a back door to state. No matter how co-located, a client gets exactly the API its role is granted, and there is still only one writer, so the audit trail is unbroken.

A consequence the rest of the design leans on: anything attributed to "the orchestration layer" — minting sessions, stamping turns with the time, turning `ScheduledJobFired` into action, composing briefs — is server-side, not a client. Clients deliver and receive; the server owns scheduling, time, and memory. No scheduler belongs inside the Discord adapter.
