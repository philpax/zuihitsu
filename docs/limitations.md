# Known limitations and recorded decisions

This document is the register of the system's accepted residual risks and its recorded decisions. A residual risk is a live property of the current deployment that carries a known hazard we accept; a recorded decision is something deliberately not built, noted so its absence reads as a choice rather than an oversight. Every item anchors in present behaviour — what the system does, and when to revisit — and actionable work lives in the issue tracker rather than here.

## Named residual risks

- *The third-party residual is judgement, not mechanism.* `PrivateToTeller` is teller-gated; the subject is mechanically protected and named excludees are mechanically filtered, but a co-present *unnamed* third party is governed by agent judgement plus the inline marker. The exposure is bounded in system state (no durable artefact, never synthesised into prose, does not replay) but not bounded in the world — an aired confidence cannot be retracted. If third-party judgement proves unreliable, two escalation levers exist but are not built: a confidence tier could be made to suppress for non-subject third parties (teller-only with opt-in sharing), and `told_in` — currently recorded as provenance only, unread by the predicate — could gate context leakage; the data is already recorded, so the lever is available the moment cross-context leakage proves real.

- *Exclude protection depends on agent recall.* The read path honours `Exclude(set)` exactly: a named excludee is mechanically filtered, and no entry surfaces to them. A write surface does not exist (#32) — no agent, operator, or adjudication path constructs an `Exclude` set today. Once one does, protection still depends on the agent recalling to name the implicated third party at write time; the mechanism enforces exactly what it is given, but cannot guarantee the recall that gives it.

- *Non-person memories have no subject-guard.* `PrivateToTeller` on `project/*`, `topic/*`, and the like is teller-gated only; protecting a specific party would require `Exclude`. A sensitive non-person memory is therefore protected only if write-time sensitivity inference fires — pure judgement, no structural backstop (the `sensitive_non_person_marked` scenario in `crates/eval/src/scenarios/privacy.rs` probes the rate). A deliberate asymmetry; if the target model marks these unreliably, a backstop such as an operator-set sensitivity default per namespace or per context is the lever, and the reply lane — the eval harness's real-model scenario runs — is where the data to decide comes from.

- *`#confidential` removal is retroactively visible.* The teller-private marker resolves a room's `#confidential` flag at brief-build time, so removing the tag silently changes how *historical* asides told in that room render thereafter — they stop being marked confidential. This is intended (the room is no longer confidential, and the predicate is unchanged either way — only the marker's *strength* shifts, never whether an entry surfaces), but it is the same retroactive-visibility shape as `same_as` unmerge: an action changes how past confidences are treated and cannot un-change anything already disclosed under the old understanding. Removal is operator-only (see [Trust model](trust-and-authority.md#trust-model)), so it cannot be steered from a platform conversation; the shape remains for the operator's own removals. Named as that family so it reads as a known property, not a bug.

- *`same_as` unmerge is irreversible in the world.* An erroneous merge has *already authorised disclosures* across the wrongly-unified class, and removing the link cannot retract them — the retroactive-visibility residual survives whatever mechanism removes the edge. Operators will sometimes merge wrongly, so this is not hypothetical. The mechanism work (graph-closure recompute plus retroactive-visibility accounting) is #31.

- *Volatility manipulation is judgement-guarded.* `set_volatility` carries no authority gate, so a participant can steer the agent into lowering another memory's volatility, and the entry then stops reading `[stale]` past its horizon. Accepted as a residual: the staleness marker is a hedge, not a gate — it changes how the agent qualifies a fact, never whether an entry surfaces — and gating it would take routine reclassification away from the agent. The two mutations with retroactive or suppressive reach, `#confidential` removal and cross-participant supersession, are structurally gated instead (see [Trust model](trust-and-authority.md#trust-model)).

- *Paraphrase-aware leak matching.* Reply-surface leak detection cannot be a substring check — a real model paraphrases a confidence ("on his way out" for "being managed out"). The matcher is a per-scenario judgement, and a too-narrow one silently passes a real leak, which is the worst failure mode for the most important tests. Seed the matchers from actual target-model outputs once the reply lane runs, and treat the matcher itself as something to review, not trust.

- *External I/O and its egress surface.* The agent reaches outward only through operator-configured MCP servers, and the fetch happens *in* the server, so the egress surface lives there: a network-capable server must enforce egress blocks (resolved-IP blocks for link-local, loopback, RFC1918, and IPv6 unique-local ranges) in its own launch config, where such flags live. What is still missing before *untrusted* exposure is a URL-allowlist or capability layer over which servers and tools a given participant may invoke (the agent's MCP tools are available in every block, including participant-facing ones — the per-server `allow` and `deny` lists narrow the surface but are operator config, not a per-participant grant). Accepted for the trusted-operator, loopback-only target; it must close before the agent is exposed to untrusted participants.

- *External I/O is not replay-faithful.* MCP tool results are not captured in the log (see [Lua API → External I/O via MCP](agent-loop.md#external-io-via-mcp)). A block's *effects* replay faithfully, so state is always reconstructible, but regenerative replay re-runs the call against a world that may have changed, and the audit trail cannot show the exact content the agent read. This is inherent to external I/O, not a deferral; the only way to "fix" it would be to record every fetched page, which is neither faithful (the live page still drifts) nor desirable. A second, related consequence: a block that has performed external I/O cannot be transparently retried on a lock-timeout abort (the effect already happened and cannot be rolled back), so the timeout surfaces to the agent as an error rather than a silent retry, and a side-effecting tool sequence (a multi-step form fill) interrupted partway is the agent's to notice and recover. The mechanism guarantees atomicity for *log effects*, not for the outside world.

- *Control endpoint trust breaks behind a reverse proxy.* The control API's authentication model — loopback-trusted, remote-keyed, fail-closed, per-surface key arrays — is owned by [Clients and the server boundary](trust-and-authority.md#clients-and-the-server-boundary). The residual is environmental: trusting a peer by its loopback address breaks behind a reverse proxy, which makes every peer appear loopback, so the server must not be fronted by one without re-checking auth. A keyed *remote* CLI target is a thin follow-up that is not built.

- *Storage-layer corruption.* Event sourcing's whole promise is "rebuild from the log," so the log and its derivations carry a deterministic pressure-test suite covering partial-batch atomicity, corrupt-log detection, deterministic replay (identical graph fingerprint across rebuilds), and snapshot-plus-tail equivalence to a full replay from `seq 0`. Two structural failure modes are covered rather than tested by timing: WAL-checkpoint interleaving by SQLite's WAL design (a checkpoint never exposes uncommitted frames), and a mid-transaction snapshot by the lock discipline (commits and `snapshot_into` serialise on the graph mutex, and `graph_head` is set only at materialisation completion, so a snapshot is always taken at a clean `seq` boundary). Revisit if a corruption mode outside the suite's structural coverage surfaces.

- *Materialiser logic bugs survive replay.* The structural sibling of corruption: a wrong per-variant handler yields a clean graph reflecting a wrong interpretation, and rebuild reproduces it. There is no storage-level defence — the eval harness against materialised state is the only backstop (see [Storage → Materialised graph](events-and-storage.md#materialised-graph) and [Validation](observability-and-testing.md#validation-and-the-eval-harness)). Treat handler changes, especially to visibility-relevant events, with the same care as schema migrations.

- *Session-gap threshold.* The session-open heuristic ("first activity, or activity resuming after a quiet period") is the one place the otherwise-explicit conversation model reintroduces a timeout. Too short and briefs re-freeze and thrash the prefix cache; too long and "now" and `recent_facts` go stale within a session. The threshold is a tuning knob found against real traffic (the console's event view shows the segmentation directly). One implementation constraint, not itself open: the boundary is *recorded* (`SessionStarted` lands in the log) and *not recomputed* at replay, so tuning the threshold changes only future segmentation and never re-segments history.

- *Compaction seam.* Token-triggered re-segmentation bounds the live buffer (see [Compaction](conversations-and-briefs.md#compaction-token-triggered-re-segmentation)), but a hard cut has a cost the carryover only partly covers. What survives: anything written to memory, anything the session *referenced* (recoverable from its `LuaExecuted` events), and the raw last turns up to the carryover budget. What is lost from context (not from the log) is the ambient transcript — said but never recorded, referenced, or flushed, and older than the budget — which is the right loss. The genuine residual even *with* the flush is in-flight reasoning: synthesis the agent was mid-way through that never became a memory or a turn. A hard cut loses it. The flush helps only insofar as the agent can dump working state to memory in that one turn. The flush itself runs the model on the hot path at maximum context (worst-case latency), which the budget-gate mitigates by skipping low-activity sessions. The compaction token budget and the idle-gap threshold are jointly tuned against the same prefix-cache-thrash concern. Two eval scenarios cover the cut with deliberately different authority: `flush_preserves_visibility` gates as a safety oracle, while `compaction_preserves_working_state` is a tracked quality metric that informs tuning but does not gate.

- *Recurring materialisation interprets a narrow RFC-5545 subset.* The wake-up arming mechanism is implemented (see [Scheduled work](time.md#scheduled-work)); the residual is the breadth of the rule grammar it honours. Instance computation interprets `FREQ` (`DAILY`, `WEEKLY`, `MONTHLY`, `YEARLY`) and `INTERVAL` only — `BYDAY` is parsed but not honoured (instances fall on the anchor's weekday rather than the named one — fine when the anchor already lands there, off otherwise), and `COUNT` and `UNTIL` are absent. A malformed or unsupported rule never fires, so a mis-specified recurrence fails silently. Revisit if real usage needs the by-part grammar.

- *Cache behaviour under load.* A waiting conversation turn now dispatches ahead of queued background passes at the shared generation model: the arbiter above the model client orders turn over background and yields background work to a pending turn (see [Model sharing and the backend](write-path.md#model-sharing-and-the-backend)). What stays unmeasured is the layer below it — the backend reusing a warm prefix for conversation turns and managing its own KV-cache underneath. Under many concurrent streams plus background regeneration, prefix-cache eviction and recompute cost are real and unmeasured; the stable-prefix discipline is the mitigation, but its effectiveness needs measuring before the stream-count limit is set — and on a backend that serialises rather than batches, the contention barely arises. The *timer-driven* idle and checkpoint flush sweeps, the adjudicator, and link inference yield to conversation without a staleness escape of their own, so on a saturated instance their work defers indefinitely; only the describer, which backs a reader-visible freshness horizon, escalates past the yield once its backlog ages out (see [Freshness before a brief](write-path.md#freshness-before-a-brief)). The *session-open* checkpoint flush is the exception among the flushes: it runs inline in `ensure_session` on the opening turn's own model handle, at turn priority, so it does not yield and cannot defer — the new conversation's turn deliberately waits on it.

- *Identity continuity across model swaps.* The local model is the agent's voice and will be replaced. Memories, descriptions, and persona survive a swap; learned style and disposition shift. `produced_by` correlates behaviour changes to model versions, but how disruptive a swap feels is an open, ongoing problem, watched rather than solved.

- *Organic dynamics over time.* Whether descriptions converge or drift, whether the graph densifies usefully or sprawls, over thousands of conversations — not reachable by static review of the mechanisms. Worth watching with a long-running toy deployment.

## Recorded decisions

- *No platform connectors are built.* No named platform adapters or external connectors exist (Discord, Slack, or others). The client surface is the web console's direct interface and the CLI's `send` and `join`; both normalise deep links to canonical `[turn:<id>]` tokens, so the connector contract is honoured, but no connector implements it. The Discord connector is tracked at #28.

- *A relationship-aware disclosure layer is not built.* The system does not bias disclosure by whether the present audience should hear about the subject's affairs; it needs a who-should-know-what model the system does not carry. Revisit if relationship-scoped disclosure becomes a real need rather than a hypothetical one.

- *A relation-keyed description scope is not built.* Richer summaries for trusted intimates of the subject would each reintroduce a write-time compartmentalisation boundary, so the system keeps a single description scope. Revisit only alongside a decision to reintroduce that boundary deliberately.

- *An explicit-allowlist lock-down visibility variant is omitted.* Private group chats do not require one, so the visibility postures stop short of an allowlist lock-down. Revisit if a deployment needs a chat whose audience is an explicit allowlist rather than its present set.

- *Per-block brief token truncation is not enforced.* `brief.token_budget` (default 2000) is advisory, not a hard per-block cut (#33), and brief relationships are neither ranked nor capped (#34). The deterministic ranking is fast; a participant-set-hash cache is the lever if brief composition becomes a bottleneck at conversation start (see [Size budget](conversations-and-briefs.md#size-budget)).

- *The wake-up surface drains only at session open.* A mid-session wake-up is recorded on time but surfaces at the next session boundary rather than per turn; a `surfaced_at` marker would make a per-turn drain idempotent, but it is not built. Revisit if within-session latency on wake-ups matters.

- *No periodic background jobs are configured.* The scheduler mechanism is generic over job kinds, but event-attached wake-ups are the only species in use. Revisit when a genuine periodic job (a digest, a sweep) is wanted.

- *True proactive contact is deliberately unsolved.* Initiating toward a human unprompted is not built; the turn schema already distinguishes `Initiated` from `Responding`, so the seam is present, but the design space is open at #20.

- *Within-block I/O concurrency does not exist.* Tool calls are sequential, and a block's MCP calls are synchronous with no promise API. Revisit if a block's latency profile makes concurrent I/O worth the determinism and replay complexity it would add.

- *MCP resource projection is not built.* MCP integration surfaces tools only, not resources, and MCP servers support only the stdio transport. Revisit if a server worth integrating exposes state through resources or needs a non-stdio transport.

- *The operator Lua console cannot query raw or historical state.* There is no `events.*` raw-event namespace, and the console cannot query historical (pre-head) state — it reads the current materialised graph. The Events view also cannot filter by `source` (#40). Revisit if operator debugging needs time-travel or raw-event access.

- *No whole-log regenerative replay is built.* The recorded `produced_by` provenance is not consumed by any operation that re-runs past inference across the whole log. The provenance is there for correlation and audit; a regenerative-replay pass over it is a deliberate non-goal at this scale.

- *Embedding serving is swappable, and the target embedder is unverified.* The target embedder is `jina-embeddings-v5-text-small`, served as its own OpenAI-compatible endpoint separate from generation (see [Inference and embedding backends](write-path.md#inference-and-embedding-backends)), so the two never contend. The chosen serving stack must be verified to support v5 — it is recent and may need a build accounting for its architecture and Matryoshka specifics; until then a `jina` v3/v4-small or `bge`-class model is a drop-in stopgap, since the embedder is swappable. The vector store starts as `sqlite-vec` (one process, plausibly enough for a personal agent); swap to an external store if it does not hold.

- *Embedding-model swap has a boot-time re-embed, with no zero-downtime cutover.* `EmbeddingModelChanged` invalidates every stored vector (cosine across two embedding spaces is silently wrong), so the boot path clears the vector index and rebuilds it from the log under the new model before serving — a startup downtime proportional to log size (see [Vector store](events-and-storage.md#vector-store)). The zero-downtime path (building the new index alongside the live one and cutting over atomically) is deferred: it buys nothing at the personal-agent scale, where a boot-time rebuild is tolerable and swaps are rare. Re-embedding from the log is the most expensive operation in the system; price it before relying on "rebuildable."

- *Snapshot cadence is untuned.* The storage-cost-versus-replay-cost trade is left at its default rather than tuned against realistic volume. Revisit by measuring under real load.

- *The event log grows unbounded.* There is no compaction; the log grows without bound. Acceptable at the personal-agent target scale, where the snapshot mechanism already bounds *replay* cost even as the log lengthens. Revisit (log compaction, cold-segment archival) only if growth becomes a real operational cost.

- *The event log is single-writer, not distributed.* The `Store` seam makes the backend swappable within its total-order invariant. Crossing that invariant — multiple nodes appending concurrently and reconciling — is not a backend swap: it replaces the total order over `seq` with a partial order plus merge (CRDT-style convergence or consensus, conflict resolution for concurrent appends, and a materialiser deterministic over a *merge* rather than a *sequence*). It collides specifically with the order-sensitive, stateful subsystems — belief arbitration, `same_as` class closure, and the visibility predicate evaluated against current state — so it is a real design project, not a config change. Filed so the door is open and the cost is honest.

- *Tamper-evidence is unaddressed.* The event log is the audit trail, but tamper-*evident* only with linkage: a SHA-256 chain over `seq` at the `Store` seam would make silent history-rewriting detectable. It is nearly free at the seam and painful to retrofit, but it buys nothing under the current trust model (the operator is trusted and holds the machine), so it is deferred — worth doing early *if* the trust model ever admits an untrusted operator. Adversarial operators are out of scope.

- *Read-your-writes for link readers is deferred.* The link readers — `mem:outgoing`, `mem:incoming`, and `mem:links` — reflect only committed state: a link created or removed earlier in the same block is not yet visible to a later read within that block, unlike the content-entry reads, which overlay the block's own pending appends. Relation *registration* is the one exception (a relation registered in the block is recognised by a later `link` in the same block), but the edges themselves stay committed-only. Read-your-writes over freshly written links would mean threading a pending-link overlay through the class-traversal readers, and no current practice needs a block to read back a link it just wrote. Revisit if a within-block link-then-read pattern emerges.

- *Console fold at live scale is unbounded.* The console is a read replica that folds the whole event log through the real materialiser compiled to WASM (see [Observability](observability-and-testing.md#observability)), which is what makes it a faithful debugger rather than a second opinion about what the events imply. Whole-fold is kept for correctness; the scale mitigations — snapshot catch-up, a leaner event stream, and windowing — are deferred. A graph snapshot alone does not bound the bytes, because the event stream is the bulk of what a long-lived deployment accumulates; bounding live-console memory is a real design item, deferred at the personal-agent target.

- *`LinkTypeChanged` migration is unresolved.* Whether to auto-resolve existing edges to the most recent type or flag them for manual review is undecided; the system does neither automatically yet, and link relations are otherwise uneditable (#42). The default lean is to flag rather than silently rewrite.

- *Time zones are stored as UTC and rendered contextually.* The system stores UTC and renders in context; each participant's zone is probably a fact on their `person/*` memory rather than a system-level setting. Revisit if per-participant zone rendering needs a dedicated mechanism.

- *`BeforeAfter` anchor resolution is unbuilt for vague anchors.* Anchors may point at memories whose own `occurred_at` is vague, and resolution must handle "before a thing that happened around 2019" without exploding into uncertainty, including an explicitly soft-deleted anchor. The resolution logic for these cases is not built. Revisit when relative-anchor temporal queries become load-bearing.

- *Memory granularity is left to emerge.* Nothing forces a grain (`topic/cooking` versus `topic/cooking/sourdough`); it is left to emerge. If an abstraction capability is added later that emits `concept/*` memories, those are themselves eligible for further abstraction, so a depth cap or prune policy should be chosen deliberately rather than emerged toward.
