# The write path

The write path's off-hot-path passes all run on the generation backend, so this doc starts with the two model backends, then the regeneration, arbitration, and inference passes they drive, then the concurrency discipline around them.

## Inference and embedding backends

zuihitsu is bring-your-own backend. Two models sit behind the agent. A **generation model** serves conversation replies and every off-hot-path pass that needs a model — the description regeneration, temporal extraction, and belief arbitration of [Write path and regeneration](#write-path-and-regeneration), plus merge adjudication (see [Cross-platform identity](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated)) and link inference. A separate **embedding model** produces the entry and description vectors of the vector store. Each is reached as an **OpenAI-compatible endpoint**, configured independently (`[model]` and `[embedding]` in the instance config); they may be one server or two, local or remote. The generation endpoint's model is named by `model.llm`, and the operator states its context window (`model.context_length`, which the OpenAI-style API does not report) so the agent can derive its compaction budget from a fraction of it. The spec fixes neither the models nor the serving software, and both clients sit behind the model-client seam (see [Testability](observability-and-testing.md#testability-and-abstraction-boundaries)), so nothing above the client depends on the choice.

One property is left to the deployment, because it varies by backend: whether the generation endpoint **batches concurrent requests or serialises them on a single slot**. The concurrency and scheduling discipline below is written to hold either way — it governs request *priority* (conversation ahead of background work), not the backend's internal batching.

### Transport resilience

The served generation client sits behind a resilience wrapper (operational config, `[model.resilience]`). A transient backend failure — a connect or transport error, a timeout, an HTTP 408/429/5xx — is retried with bounded exponential backoff and jitter, while a non-transient failure (a schema, auth, or other 4xx error) surfaces at once and is never retried. A circuit breaker holds the backend's health across every caller: after a run of consecutive transient failures the circuit opens and model-needing calls fail fast without touching the backend, until a single half-open probe decides whether to close it again. These retries are infra-transparent — they emit nothing to the log — so replay never depends on the policy, which is operational, not behavioural.

When the backend stays unreachable — retries exhausted or the circuit open — a routed turn does not error; it **defers**. The inbound `ConversationTurn` is already durable and no agent turn is recorded, so the next successful turn replays the deferred inbound in its buffer and one response cycle covers the backlog. Recovery is passive by design: there is no on-recovery push, because a reply has no delivery path to a platform client outside the message-response call (see [Agent-initiated speech](time.md#agent-initiated-speech)).

## Write path and regeneration

Recording a memory triggers model work: description regeneration and temporal extraction, plus belief arbitration on conflict and link inference over the new content. All of it runs on the **generation model** — the same one serving conversation (see [Inference and embedding backends](#inference-and-embedding-backends)), never the embedder. Firing it inline per `append` is a latency trap; two rules avoid it: coalesce a memory's touches into a single regeneration, and run that regeneration off the turn's hot path as a catch-up.

### Coalesce, then regenerate once

Appends are batched: a memory touched several times — within a turn, or across the window a catch-up pass spans — is regenerated once over its full current content set, not once per entry. Regenerating one stale memory issues up to three focused, schema-constrained model calls, not one, each with its own concern:

- A **combined describe-and-extract call** over the memory's [`Public`](visibility.md#visibility-versus-disclosure-and-three-postures) entries. It synthesises the `description` and, in the same reply, extracts the occurrence times of the public time-bearing statements (see [Time](time.md) → in the same pass). Synthesising over the public class only is what keeps a private aside — or an attributed fact's "via <teller>" framing — out of the always-visible summary.
- A separate **focused arbitration call**, run only when the memory has at least two entries in the wider `Public` + [`Attributed`](visibility.md#visibility-versus-disclosure-and-three-postures) slice. It scans that wider slice, numbered in one pool, so two relayed-but-conflicting accounts the agent marked `Attributed` still collide, and a public account and an attributed one of the same fact collide across postures. It is split out of the description call deliberately: bundled as a rider on the mandatory description rewrite, the conditional contradiction check gets crowded out and skipped; split out, the check is the reply's whole job. When it finds a genuine contradiction — at least two distinct competing entries and a one-line reconciling note — it emits `BeliefArbitrated`; the resolution credits the side(s) it judges correct, or credits neither when the disagreement stands and both accounts are kept.
- An **extraction-only side-pass** over the memory's private untimed entries, run only when such entries exist. A private reminder must still fire, so its occurrence is resolved, but this pass never feeds the description. For an all-public memory (the common case) this side-pass is empty; only a memory mixing public and private new content pays for it.

The arbitration is not only a logged audit record: when its resolution credits neither side, the [materialiser](events-and-storage.md#materialised-graph) projects the competing entries so a later read renders them `disputed` (the `[disputed · …]` marker and `entry.disputed` field, see [Lua API](agent-loop.md#lua-api)). That closes the loop: synthesis detects the conflict once and records it structurally, and every subsequent read of those entries carries the contested status, so the agent surfaces the disagreement instead of re-deciding it or asserting one side. The marker is derived from the *latest* arbitration per memory and the live competing set, so crediting a side or superseding one account clears it without a separate event.

The structured replies are parsed leniently: a well-formed `description` is salvaged even when an occurrence field is mis-shaped (a smaller model often gets the prose right while flattening a nested time), and a both-stand arbitration verdict parses cleanly whether the model omits `credited`, emits `null`, or sends an empty list.

### Regenerate off the hot path, as a catch-up

Regeneration and extraction never run inline in a turn or the pre-compaction flush — the conversational reply is never held waiting on summarisation, and neither is the flush. They run as a **catch-up** driven by the graph's own per-memory watermarks, the same shape as the background indexer (see [Storage → Vector store](events-and-storage.md#vector-store)), but with no separate cursor: each memory carries a `last_content_seq` (bumped by every content event) and a `last_described_seq` (advanced by the `DescribePassCompleted` event the pass emits after considering it), and a memory is stale — owed a pass — exactly while `last_content_seq > last_described_seq`. Because both watermarks are materialised from the log, the describe backlog is derived state that survives a restart rather than an in-memory cursor reset at every boot. A crash simply re-derives the pending set from the log.

The pass walks the stale set oldest-content-first, and for each memory regenerates its description, arbitrates its beliefs, and resolves any occurrence it left untimed, then commits the synthesis events and a `DescribePassCompleted` in one batch and materialises. The `DescribePassCompleted` advances the watermark past the memory's content **whether or not synthesis produced anything** — a memory with no public entries is still marked considered, so a persistent failure to describe it does not wedge the backlog. It is idempotent and resumable: a memory already fresh is skipped, so an idle tick is cheap, and re-running reproduces the same events. The pass holds its serialisation guard **per memory**, not across the whole pass, and re-checks staleness under the guard each iteration, so a narrow session-open pass interleaves with a long background backlog instead of waiting behind it, and two passes never redescribe the same memory for the same change.

The non-served paths — tests, the reply-lane eval harness, one-shot tooling — drive the *same* catch-up explicitly (a `describe_catch_up` call, or the narrowed `describe_catch_up_for`), exactly as they drive the indexer, so a synchronous caller forces regeneration to a known point and then reads fresh descriptions. A memory written this turn is readable as raw content entries immediately (cheap inserts); only its synthesised description, arbitration, and extracted times lag until the next catch-up — acceptable, since the raw entries are already readable and only the derived synthesis trails them. The one place the lag is not acceptable is a brief about to be composed off those descriptions, which the forcing below closes.

### Background synthesis runs off the hot path on timers

The background passes — the describer, adjudicator, and link-inference pass, plus the vector indexer and the scheduler driver — each run on their own short timer in the served runtime, independent of conversation load. A turn commits its entries and replies without waiting on any of them; the passes catch the derived state up afterward.

What the agent does control is the *shape* of the prompt it sends, so the serving layer's own prefix cache stays warm across a session. The frozen system prompt is deliberately the cache-stable prefix — which is why a join arrives as a suffixed message rather than a prompt rebuild, and why the per-turn time rides in the buffer rather than the prompt (the conversation-boundaries principle, see [Architectural principles](overview.md#architectural-principles), and [Time](time.md)) — so a conversation turn reuses a warm prefix instead of recomputing it. A background pass needs no checkpoint of its own: re-running a regeneration from scratch is cheap and idempotent, and the backend manages its own KV-cache and prefix reuse underneath.

### Freshness before a brief

Because the describer runs on a timer, a memory written moments ago may not yet be described when it is next read. For most reads that lag is acceptable — the raw entries are the truth, the description a convenience — but a brief composed off descriptions must not read stale prose. So **composing a brief forces the describe catch-up** for exactly the memories that brief is about, and no more: the joiner's brief on a [mid-session join](conversations-and-briefs.md#mid-conversation-joins), and the working-set memories the pre-compaction flush just wrote before the post-compaction session's brief (see [Conversations and contexts → Compaction](conversations-and-briefs.md#compaction-token-triggered-re-segmentation)). The narrowed pass (`describe_catch_up_for`) describes only the stale memories among the brief's read set, leaving the rest of the backlog to the background timer — no skip, no redundancy. Because the brief is frozen per session, the catch-up only has to be current at the moments a brief is composed, not on every turn, so a brief is never built off stale prose regardless of backlog.

### Link inference

A cursor-driven off-hot-path catch-up, `link_inference`, mirrors the describer and adjudicator in shape — a pass over the log window since its cursor, gated by its own `link-inference` prompt template (no template registered, no pass) — but extracts *relationships* rather than descriptions or merge verdicts. For each memory whose content changed since the last pass, it reads the memory's `Public` entries (private asides never enter the prompt here), its existing links (to avoid duplicates), and the registered relations (to reuse before coining). It then asks the model, in one schema-constrained call, to identify relationships implicit in the content that link the memory to one of the candidate target memories. Each relationship is written as a literal subject–relation–object sentence ("`person/theo` `mentored_by` `person/clara`"), so the direction is carried by the sentence itself rather than by a separate flag — a direction field detached from the sentence inverts asymmetric links at scale, so the sentence carries the direction here just as it does for `links.create`. Exactly one of the subject and the object must be the memory under inference; the other is a candidate's handle. The candidate list is capped at 100 memories to bound the prompt — a target the model names beyond the cap is still resolved by handle, so a memory left out of the candidates is not lost.

New relation types are registered (`LinkTypeRegistered`, an upsert) before the links of that type are created, with the model's proposed cardinalities parsed and an unparseable one dropped; named entities are resolved to existing memories via their handles (the model never invents ids — an unresolved entity is skipped, not minted); and a sentence that does not name the memory under inference on exactly one end (naming it on neither or both), or that carries a relation neither registered nor proposed, is dropped. Each considered memory emits a `LinksInferred` event carrying the pass's result and its provenance, even when nothing was inferred, so the pass's coverage is auditable. Links the pass creates carry `source: Inferred` and no teller, distinguishing them from agent-, operator-, and adjudicated-authored edges. Idempotent: a duplicate `LinkCreated` is a graph no-op (`INSERT OR IGNORE`), a `LinkTypeRegistered` is an upsert, and the cursor advance keeps a window from being re-scanned — the cursor advances to log-head whether or not the template is registered, so a toggled-off pass does not re-scan later.

The pass does not infer `same_as` (those flow through the adjudication gate, not here) and does not consider private entries. Each `LinkCreated` carries a visibility posture, and an inferred link inherits the visibility of the source entry it was extracted from (see [Visibility](visibility.md) → inferred links); since the pass reads only `Public` entries, its links are `Public` in practice, and the prompt steers toward structural or neutral relationships whose public surfacing is safe.

## Concurrency

Multiple conversations may be in flight at once (a direct-interface DM and a live Discord group), subject to a configurable stream-count limit, since the shared generation model is the binding constraint. "In flight" here means concurrent at the *logic* layer — overlapping blocks, locks, and turns — independent of whether the backend batches those turns at the model or serialises them (see [Model sharing](#model-sharing-and-the-backend) below). Two disciplines govern interaction.

### Per-memory mutual exclusion, scoped to in-flight blocks

When a `LuaExecuted` block in conversation A has read or written a memory, concurrent reads or writes to that memory from conversation B block until A's block finishes. The mutex granularity is the memory, not the conversation; its lifetime is the code block, not the turn. A long turn in A doesn't block all of B, only B's operations against memories A actually touched.

### Class-wide locking on traversing reads

A *traversing* read (any agent-facing operation that auto-traverses `same_as`) locks the full equivalence class of the queried memory, not just the queried stub, because otherwise a concurrent write to a sibling stub could produce a torn merged read. This is live for the operator's own merged identity: reading "you" in a Discord conversation while a DM or scheduled job touches your direct-interface stub spans both stubs. Writes are not traversed, so a write locks only its target stub. Synthesis traverses the `same_as` class (see [Visibility](visibility.md)) and so locks the full class, which for a singleton class is just the one stub.

### Lock acquisition: timeout-and-retry, not an ordering protocol

At the deployment's scale (a few concurrent streams, small `same_as` classes), the binding risk isn't true lock cycles but a block holding a lock across slow I/O — `dave:append(...)` then a slow `mcp.*` call parking while another conversation waits on `dave`. A per-block duration timeout is the backstop (`block_timeout_seconds`, set generously above a single MCP call's own timeout so an ordinary multi-call block is never cut): a block held too long, whether on a genuine wait cycle or on slow I/O while holding locks, aborts, releases its locks, and retries, bounded by `max_block_attempts`. This is safe because blocks are atomic transactions — see [Lua API → Block transactionality](agent-loop.md#block-transactionality) — an aborted block has emitted nothing, so the retry is the only observable trace. A retried block re-runs the Lua from scratch; the VM globals (the scratchpad) persist and are not transactional, so a non-idempotent scratchpad write is observable across attempts.

The one exception is external I/O: a block that has already made an MCP call has caused an effect that cannot be rolled back, so it is not silently retried, and the timeout surfaces as a catchable error instead (see [Lua API → External I/O via MCP](agent-loop.md#external-io-via-mcp)). An elaborate ordering protocol (ULID-ascending acquisition, wait-for-cycle detection) is unnecessary at this scale and is deliberately omitted; the timeout is sufficient.

One pattern is worth naming, because it can hit the timeout by design: *mutate-then-traversing-reread*, as in `dave@discord:append(...)` then `dave:get()` to see the merged view, where the traversing read grows the lock set to the full class and may contend with a sibling-stub writer. Lock acquisition is entirely automatic — each traversing read locks its class on the way through, and a block cannot pre-declare its lock set — so the block simply accepts the cheap abort-and-retry when it loses the race.

### Model sharing and the backend

How concurrent turns share the generation model is the backend's affair, and deployments differ: some batch concurrent requests, some serialise them on a single slot. The agent is written not to depend on which. The one limit the agent imposes on *turns* is the stream-count semaphore (`max_concurrent_streams`, default 4): a permit is held for each in-flight inbound message's whole handling — the turn and any compaction flush it triggers — so no more than that many conversation turns crowd the shared model at once, and further streams queue for a permit. A model-driving control action (an [operator imprint](lifecycle.md#imprint-interview-creator-self-introduction), a merge confirmation's re-brief) takes a permit the same way.

Turn-over-background **priority** is arbitrated one layer above the shared client, so a waiting conversation turn dispatches ahead of queued background synthesis. Every caller reaches the model through a small arbiter that hands out two handles over the one client: the conversation path holds a *turn* handle, and each background pass — describer, adjudicator, link inference, and the idle and checkpoint flush sweeps — holds a *background* handle. The priority a call carries is the handle's, not an argument, so a background pass cannot dispatch at turn priority by mistake. A background call yields the model while any turn is at it — before dispatching, it waits until no turn-priority call is queued or in flight — then runs to completion once it starts. The discipline is **ordering, not preemption**: an in-flight generation is never cancelled, and a turn arriving mid-call waits only for that one call, not for the queue of background work behind it. Priority is scoped to each model call rather than the whole turn: a turn doing Lua work between its model calls holds no marker, so background work runs in that gap instead of stalling behind a turn that is not actually at the model — which is what lets the arbiter and the stream semaphore compose without a mid-turn turn starving background. The indexer is not in this arbitration: it drives the embedder, not the generation model. If the backend serialises, background work simply takes the slot whenever no turn wants it; if it batches, turns run together up to the backend's own concurrency and background fills the remainder. Either way nothing below the arbiter changes. The embedder is never in this contention: it is a separate endpoint, so retrieval embedding and generation do not compete for the same resource.
