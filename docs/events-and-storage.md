# Events and storage

The event log is the source of truth; the graph, vector index, and snapshots are derived projections rebuilt from it, feeding visibility, briefs, and observability.

## Event sourcing

All state changes are events, and graph state is a pure projection of them: nothing mutates the running state except by appending an event to the log.

Every payload is tagged by a `type` discriminant on the wire (`#[serde(tag = "type")]`), and the catalogue grows deliberately: a new capability adds a new variant or a new field, and old logs replay unchanged. The serialised payload carries no version field — the `Event` envelope is only `seq`, `recorded_at`, and the `payload` itself. Field-level evolution is handled with `#[serde(default)]` on each added field (so a shorter historical payload deserialises with the field's default) and `#[serde(alias = "…")]` on renamed variants (so a log written under an old name still reads). (Where the prose below says "version-1 payload", it means a historical payload that lacked a field now present, for which serde fills the default.)

The events are not `Eq`: `ConfigSet` carries a `Settings` snapshot whose search weights are `f32`, so equality is `PartialEq` throughout.

**Event types:**

- `GenesisCompleted { manifest_hash, template_versions }` — marks a completed [genesis](lifecycle.md#creation-idempotent-via-the-operator-surface) sequence. Boot branches on the presence of this event, not on log emptiness, so a partially-initialised log is distinguishable from a finished one. `manifest_hash` fixes the genesis manifest that produced the seed state; `template_versions` maps each seeded template's name to the version it was registered under, so a later run can tell whether its build's templates have moved past the ones baked in.
- `MemoryCreated { id, name }` — creates an empty memory. Any initial content (the second argument to `memory.create`, or a seed disposition entry) is recorded as a paired `MemoryContentAppended`, so there is exactly one provenance path for all content.
- `MemoryRenamed { id, old_name, new_name }` — changes a memory's agent-facing handle, leaving the ULID and every relational reference untouched (see [Identity → Renaming](data-model.md#renaming-the-same-memory-a-new-handle)). `old_name` is kept for the log and the console, never re-surfaced in a read, so a renamed person's prior name (a deadname, in the case it most matters for) does not resurface.
- `MemoryDeleted { id }` — a soft delete: the contents are preserved for replay and audit, and the projection sets a deletion flag rather than dropping anything.
- `MemoryContentAppended { id, entry_id, asserted_at, occurred_at, text, told_by, told_in, visibility }` — records one content entry. `asserted_at` is when the agent recorded the fact; `occurred_at` is the optional real-world time the fact is *about* (see [Time → bi-temporality](time.md#bi-temporal-entries)), an `Option<TemporalRef>` carried `#[serde(default)]` so pre-time logs, which lacked the field, replay as `None`. `told_by` is the teller (a participant, the agent itself, or bootstrap); `told_in` is the `Option<ConversationRef>` (a turn or the room) it was told in, resolved to its confidentiality when contexts exist (`None` until then); `visibility` governs the read-time predicate. An entry exceeding the configured character limit (`memory.max_entry_chars`, default 1000) is rejected before it is buffered, surfacing a teachable error that guides the agent to summarise what it learned rather than pasting source content verbatim.
- `MemorySuperseded { id, entry, superseded_by }` — marks an entry superseded by a newer one: the agent corrected or retracted a fact, recording which entry replaces it (see [Visibility → superseded entries are not live](visibility.md#superseded-entries-are-not-live), **Data model → `superseded_by`**). The original `MemoryContentAppended` stays immutable; applying this stamps the superseded entry's `superseded_by`, so live surfaces exclude it while history surfaces (`mem:history()`, the console) still show it. `entry` and `superseded_by` belong to the same `same_as` class as `id`.
- `EntryTemporalResolved { id, entry_id, occurred_at, produced_by }` — the turn-end extraction pass resolved an entry's `occurred_at` from its natural-language text ("last Tuesday") into a structured `TemporalRef`. The original `MemoryContentAppended` stays immutable; applying this recomputes the entry's denormalised occurrence columns, and `produced_by` records the extracting inference. The companion to `MemoryDescriptionRegenerated` on the write-path catch-up: description and times come off the same pass for public content, and untimed private entries get a focused extraction-only side-pass (see [Write path → coalesce, then regenerate once](write-path.md#coalesce-then-regenerate-once)). A resolved occurrence is inference, not ground truth — the pass anchors relative phrases against the declared current time and can misfire on anaphora — so a stated (authored) occurrence always outranks an extracted one at read time (see [Time → Storage and resolution](time.md#storage-and-resolution)).
- `EntryTemporalResolveFailed { id, entry_id, raw, reason, produced_by }` — records that the turn-end extraction pass could not parse the model's date string for this entry. Log-only: the appended entry remains untimed, and this event surfaces the failure for operator review rather than mutating any state. `raw` is the JSON form of the extracted value for debugging, `reason` the parse failure, and `produced_by` the extracting inference. It is the failure companion to `EntryTemporalResolved`, so a pass that saw a date it could not decode leaves a durable trace instead of silently dropping it.
- `EntryDescriptionMirrored { id, entry_id }` — marks a content entry as a mirror of its memory's description (the seed entry `memory.create` appends from its `description` argument) rather than an account of a real occurrence. Applying it stamps the entry's `description_mirror` flag; the turn-end temporal extraction then skips it. A description mirror restates what the memory *is*, naming no time, so an extractor asked to time it would fabricate the conversation's "now", and that guessed date would then collide with a later, correctly-dated append on the same memory. The mark is emitted right after the seeding `MemoryContentAppended`, in the same block, so the seed entry's intent survives replay rather than being re-inferred from event adjacency (which cannot tell a seeded create from a bare create followed by an append).
- `ScheduledJobFired { entry_id, memory, fired_at }` — fires a calendared entry's wake-up when its occurrence comes due. (Such an entry's `occurred_sort` is later than its `asserted_at` — scheduled forward, not recorded after the fact — and the wake-up fires when `occurred_sort` passes `now`.) Recorded in the log so the wake-up surface is a function of the log, not a live clock; applying it stamps the entry's `fired_at`. The fired entry waits in the surface until an eligible session drains it.
- `ScheduledItemSurfaced { entry_id, memory, session, surfaced_at }` — marks a fired wake-up delivered: the drain raised it as an `Initiated` system turn in `session`, so it is never raised again. Applying it stamps the entry's `surfaced_at`.
- `MemoryDescriptionRegenerated { id, new_text, produced_by }` — replaces a memory's synthesised description; the text is produced by the model and applying it to the projection is purely mechanical. `produced_by` records the inference that wrote the description (`None` only for a hand-seeded one).
- `BeliefArbitrated { memory, competing_entries, resolution, produced_by }` — emitted by regeneration when the entries it arbitrates over conflict. `competing_entries` is the set of conflicting `EntryId`s the pass saw; `resolution` is an `ArbitrationResolution { credited, statement }` recording which entry or entries it credited (by `EntryId`) and the reconciling statement it wrote. Arbitration scans the memory's `Public` and `Attributed` entries — a wider slice than the `Public`-only entries the description is built from — so a public account and an account the agent relayed as `Attributed` can collide with each other, and this records the agent choosing between the conflicting assertions, making "why does the agent believe X" replayable instead of buried inside a description string. When the resolution credits *neither* side (an unresolved disagreement, both accounts standing), the [materialiser](#materialised-graph) projects its competing entries so a read marks them `disputed` (see [Write path → arbitration](write-path.md#coalesce-then-regenerate-once)); crediting a side, or superseding one account until fewer than two competitors remain live, clears the marker.
- `MergeProposed { from, to, source, rationale }` — a recorded judgement that two `person/*` stubs may be one human, held for adjudication (see [Cross-platform identity](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated)). Inert: not a `same_as`, not projected into the graph, so it changes no `class_id` and surfaces nothing — both stubs stay in their own classes until a `MergeAdjudicated` accepts. `source` is `Agent` (the agent's own `:propose_merge`) or `Orchestration` (raised at mint time when a platform arrival's handle matches an existing unbound, never-platform-bound stub); a payload with no `source` reads as `Agent`. `rationale` is the proposer's stated grounds, if any, which the adjudication pass weighs as the proposer's *claim*, not as evidence, against the two stubs' independently-recorded facts; `None` for an orchestration handle match or a `same_as`-via-link routed here.
- `MergeAdjudicated { from, to, accepted, rationale, produced_by }` — the adjudication pass's verdict on a proposal, weighing the two stubs' independently-recorded facts against the confidences at risk. A log-only audit record. On `accepted` the pass also authors the merging `LinkCreated { relation: "same_as", source: Adjudicated }`; on refusal the proposal stands recorded for the operator backstop. `produced_by` records the inference, so a refusal is replayable and a wrong accept is auditable.
- `LinksInferred { memory, result, produced_by }` — the off-hot-path link-inference pass's parsed result for one memory (see [Write path → link inference](write-path.md#link-inference)): the new relations the model proposed and the links it identified, recorded whether or not any were committed (a pass that found no relationships is as diagnostic as one that found the wrong ones). Log-only audit record — the materialiser ignores it, since the actual links and registrations are committed as separate `LinkCreated`/`LinkTypeRegistered` events with `source: Inferred`. `result` is a `LinkInferenceResult { new_relations, links }`: `new_relations` are relations the pass coined for a relationship no registered relation fit (`InferredRelationSpec` — name, inverse, cardinalities, symmetry, reflexivity, description), and `links` are the identified relationships (`InferredLinkSpec` — the 1-based statement number the model cited, the relation, the target, and the direction "to"/"from"). The pass's model-call deliberation is captured by the `ModelCalled` events it emits at `Full`; this event carries the structured outcome so the log shows what the model decided without reconstructing it from the raw completion.
- `DescribePassCompleted { memories }` — records one describer pass: every memory the pass considered, whether or not synthesis succeeded (matching the describer's advance-past-failure discipline). Applying it stamps each listed memory's `last_described_seq` to this event's seq, so a memory is stale exactly while its `last_content_seq` outruns its `last_described_seq` (see [Write path → regenerate off the hot path](write-path.md#regenerate-off-the-hot-path-as-a-catch-up)). The list is a `Vec` so a pass may batch several memories, though a per-memory pass records a batch of one. Log-derived state: the describe backlog survives a restart, since it is a function of the log rather than an in-memory cursor.
- `MemoryVolatilitySet { id, volatility }` — sets how sharply a memory's facts decay in search ranking; `volatility` is `Low`, `Medium` (the default), or `High` (see [Data model](data-model.md)).
- `TagCreated { name, description }` — creates a tag, which always forces a purpose. Distinct from application, which never mutates the description.
- `TagDescriptionChanged { name, new_description }` — rewrites an existing tag's purpose line.
- `TagAppliedToMemory { memory, tag }` / `TagRemovedFromMemory { memory, tag }` — attach or detach a tag on a memory; neither touches the tag's description.
- `LinkTypeRegistered { name, inverse, from_card, to_card, symmetric, reflexive, description }` — registers a relation in the schema, accessible under either label; the inverse view's cardinality is computed (the registry lives in data, not code — see [Data model](data-model.md)). `from_card` and `to_card` are `One` or `Many`. `description` is the relation's one-line purpose, surfaced in the system prompt's relation registry and in `links.list`/`get` so the agent knows which relation fits which situation.
- `LinkCreated { from, to, relation, source, told_by, told_in, visibility }` — creates a directed edge; the materialiser canonicalises direction at write time, so a link asserted under either label produces the same stored edge. `source` is `Agent`, `Operator`, `Adjudicated`, or `Inferred`: `Adjudicated` is a `same_as` authored by the merge-adjudication pass, the only path to a merge without the operator (see [Cross-platform identity](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated)), and `Inferred` marks a link the off-hot-path link-inference pass authored from a relationship implicit in content. `told_by` is the teller who asserted the relationship — captured for every block-authored link the way a content entry captures its teller, so a belief-bearing edge carries who claims it; `None` for a link with no teller behind it (the adjudicated `same_as`) and for pre-provenance logs (`#[serde(default)]`). `told_in` is `Option<ConversationRef>` (a turn or room the link was asserted in), mirroring content entries' `told_in`; `visibility` is the audience posture, governing the read-time `link_visible` predicate (see [Visibility → link visibility](visibility.md#link-visibility)) — `Public` for structural/operator/adjudicated links, `PrivateToTeller` for a participant-asserted belief about someone else, `Attributed` for a secondhand relayed relationship. Both `told_in` and `visibility` default for pre-visibility logs (`None` and `Public`). The link readers surface provenance (see [Lua API → link readers](agent-loop.md#memory-operations)) and filter through `link_visible` when an audience is present.
- `LinkRemoved { from, to, relation }` — removes a directed edge.
- `PromptTemplateRegistered { name, version, body, source }` — registers a versioned orchestration prompt template (scaffold, description-regen, temporal-extraction, flush, imprint, merge-adjudication, or link-inference). Not agent-editable; updating a template is a new registration with a bumped version. `source` is the authoring authority (`Bootstrap`, `Orchestration`, `Operator`, or `Agent`).
- `ConfigSet { settings, source }` — a whole behavioural-settings snapshot: one strongly-typed `Settings` struct grouped into substructs (compaction token budget, idle-gap threshold, flush-gating threshold, carryover character budget, brief and present-set budgets, search weights, `max_steps`, memory entry character limit, and so on). The current settings are the latest `ConfigSet`; the default snapshot is seeded at genesis, and an operator change reads-modifies-writes the whole struct. `source` is `Bootstrap` for the genesis-seeded snapshot and `Operator` for a change made through the control interface. Behavioural config lives in the log precisely so replay reproduces the behaviour that the values in force at the time produced (see [Initialisation → Configuration](lifecycle.md#configuration)).
- `EmbeddingModelChanged { from, to }` — records an embedding-model swap: the model that produced the existing vectors, `from`, gave way to a new one, `to`. Not a `ConfigSet`, since it isn't a flat behavioural knob but a logged migration; it bears only on the vector index, never the graph, and drives the boot-time re-embed described under [Vector store](#vector-store).
- `LuaExecuted { conversation, turn_id, script, result, touched, terminal_cause, duration_ms }` — records one executed Lua block. `touched` is the set of memory IDs the block read or wrote (the per-block lock set the concurrency layer already acquires — see [Concurrency](write-path.md#concurrency)), recorded so the touched set is recoverable at replay; reads emit no other trace, so this is the only durable record of what a block looked at. `duration_ms` is the block's wall-clock execution time (the final attempt's, on the retry path), for the console's turn timeline. See below and [Lua API](agent-loop.md#lua-api).
- `ModelCalled { conversation, turn_id, phase, request_digest, request, completion, reasoning, finish_reason, usage, duration_ms }` — log-only telemetry: the deliberation surface the console reconstructs (see [Observability](observability-and-testing.md#observability)). The materialiser ignores it, so replayed state is identical with or without it, while the recorded (non-deterministic) reasoning, usage, and latency are reproduced verbatim. `phase` is `Step` (a step of the agent loop) or `Synthesis` (the post-turn description synthesis). `request` is the delta-encoded `RequestRecord` at the `Full` capture level and `None` at `Digest`; the first call of a `(turn_id, phase)` group records a `Base` (the frozen request shape plus the initial message buffer) and each later call a `Continuation` (only the messages appended since). `request_digest` (a SHA-256 over the request actually sent) is always present, so a reconstructed prompt can be checked against it.
- `ConversationTurn { conversation, turn_id, role, text, participant, initiation, produced_by, brief }` — `role` is `Participant` (an inbound message), `Agent` (the agent's response, a silent terminal with empty `text`, or a surfaced `max_steps` error), or `System` (an injected join-brief or drained wake-up); `initiation` is `Responding` or `Initiated`. `participant` is the speaker of an inbound message (`None` for the agent's own and system turns). A turn carries no `session` field — its session is the enclosing `SessionStarted`/`SessionEnded` bracket. `produced_by` records the inference behind an `Agent` turn (a `Flush`-template turn is what a checkpoint's flush watermark scans for — see [Compaction](conversations-and-briefs.md#compaction-token-triggered-re-segmentation)); participant and system turns are not inference, so it is `None`. `brief` is the structured `Brief` behind a mid-session join's `System` turn, the same content its `text` carries as markup, kept as data so the console renders a proper entrance rather than raw markup. Note the vocabulary: a *turn* in the loop sense (see [Agent loop](agent-loop.md#agent-loop-and-tool-protocol)) is the agent's whole response cycle, which produces exactly one `role = Agent` event, and each inbound message is its own `role = Participant` event the loop reads.
- `ConversationStarted { id, locator, context_memory }` — opens a durable room, keyed by its `ConversationLocator`. Fires once on first contact; the room then persists across sessions for the agent's life (`ConversationEnded` fires only on permanent retirement, below). `context_memory` is the `context/*` memory minted eagerly alongside the room (see [Contexts are first-class memories](conversations-and-briefs.md#contexts-are-first-class-memories)), so the locator resolves to a first-class memory the agent can tag (`#confidential`) and reason about.
- `ConversationEnded { id }` — retires a conversation permanently. Rare — conversations are durable and long-lived; a session, below, is the bounded unit that opens and closes routinely.
- `SessionStarted { conversation, id, participants, started_at, seeded_from_turn, brief, working_set }` — opens a bounded activity window within a conversation, the brief-freeze unit. `participants` is the [present set](conversations-and-briefs.md#present-set-is-per-session-supplied-by-the-client) at open; `brief` is the composed brief block, captured here verbatim so the frozen prompt is faithfully replayable without recomposing against current state (see [System prompt → replay](conversations-and-briefs.md#on-replay-the-prompt-is-faithfully-replayable)). `seeded_from_turn` is `Option<ConversationRef>` recording the extent of raw transcript carried over when this session opened via compaction (a `ConversationRef` with the turn set; `None` for a fresh or idle-opened session) — the one carryover fact faithful replay needs, recorded rather than recomputed from the character budget. `working_set` is the working-set memory ids carried into this session from the previous segment, recorded (`#[serde(default)]`) so the brief composition is reproducible after the fact; it is empty both for sessions recorded before the field was captured and for genuinely fresh sessions, and consumers distinguish the two by the field's presence in the raw payload.
- `SessionEnded { conversation, id }` — closes a bounded activity window.
- `ParticipantJoined { conversation, session, participant, at_turn }` — a participant arriving mid-session; `at_turn` is a `ConversationRef` with the turn set.
- `ParticipantIdentified { memory, platform, platform_user_id }` — binds a `person/*` stub to a platform identity, seeding the `(platform, platform_user_id) → memory_id` operational mapping. Emitted on first contact from a platform client (alongside the `MemoryCreated` that mints the stub) and whenever an existing stub is associated with a further platform identity. The mapping is operational, not a memory-graph fact, so it lives in this event rather than as a relation (see [Identity](data-model.md#identity-and-participants)).

**`LuaExecuted` records what the agent saw.** The stored `result` is the value rendered back into the next inference step — rendered text, not a live handle — so that faithful replay feeds the model exactly the string it originally saw. A block is a transaction (see [Lua API → Block transactionality](agent-loop.md#block-transactionality)): side-effect events are buffered and emitted atomically at commit, all carrying the block's `turn_id`.

Whether a `LuaExecuted` event is emitted at all depends on whether the agent observed the outcome:

- *Agent-visible terminal outcomes* — runtime errors and explicit `block.abort(reason)`. These emit a `LuaExecuted` with `terminal_cause` populated (`TerminalCause::Error("...")` or `TerminalCause::Aborted("reason")`), because the error string or abort acknowledgement is an input to the next inference step and replay needs it. `result` is `null` unless intermediate reads were rendered back to the agent before the terminal outcome, in which case those values are captured too. The rule: `result` captures whatever the agent actually saw.
- *Infra-transparent retried outcomes* — lock-timeout aborts (see [Concurrency](write-path.md#concurrency)). These emit nothing; the retry's eventual `LuaExecuted` is the only trace, because the agent never saw the aborted attempt.

The test is whether the agent saw the outcome. If it did, the outcome is recorded; if not, the retry carries it. A block that commits normally has no `terminal_cause` at all.

**Provenance on inference.** Any event produced with model inference (`MemoryDescriptionRegenerated`, agent `ConversationTurn`s, `EntryTemporalResolved` and `EntryTemporalResolveFailed` when temporal extraction runs, `BeliefArbitrated`, `MergeAdjudicated`, `LinksInferred`, and any translated entry) carries `produced_by: ProducedBy { model_id, template_name, template_version }`. Purely mechanical events leave it `None`. This makes "which model and template wrote this" answerable retroactively and lets regenerative replay choose to trust or re-run.

**Per-memory history** is projected on demand by filtering events on target ID; cheap with an index. Exposed to the agent as `mem:history()`.

## Storage and materialisation

Three layers, distinct roles.

### Event log

Durable, append-only, the source of truth. It sits behind a `Store` seam — `append(events)`, `read_from(seq)`, `head()`, `truncate_to(seq)`, and `subscribe()` — so the backend is swappable: SQLite is the backend, and any store preserving a single total order over `seq` slots in. The SQLite backend is a database in WAL mode: one `events` table keyed by an autoincrementing `seq` (its `INTEGER PRIMARY KEY`), carrying the recording timestamp, the payload's type name, an optional `target_id` (indexed, so one memory's or one conversation's history is a cheap lookup), a `version` column stamped from the derived `payload.version()` accessor (recorded metadata, not a field in the payload JSON), and the JSON payload itself. Written once, never modified. If everything else is lost, the system rebuilds from this.

An `append` writes the whole batch inside a single SQLite transaction, assigning consecutive `seq` values and stamping every payload with one `recorded_at`. The batch is the unit of atomicity — it maps onto a block's buffered effects in the commit path — so a crash between the batch's first insert and its commit rolls the entire batch back, and a reopened log holds exactly the committed events, never a partial one. A corrupt or torn log surfaces an error rather than silently returning short or empty data, which for a system that rebuilds from the log is the worst possible failure.

One writer, many readers. Opening a file-backed log takes an exclusive advisory OS lock on the file (via `fs2`), held for the store's lifetime, so a second writer is refused while the first holds it and admitted once it is released. An operator can still open the log **read-only**, taking no lock, to inspect a running agent's log without contending with it. `truncate_to` — remove every event past a `seq`, leaving that `seq` as the new head — is the sole exception to append-only; it exists only for the operator's revert path, and it deliberately leaves the derived stores (graph, vector index, snapshots) ahead of the shortened log, so the caller is responsible for resetting them to rebuild from the shortened log on the next boot.

The total-order guarantee is not incidental: faithful replay, the materialiser, time-travel, and "the log is the agent from `seq 0`" all assume one authoritative sequence, and a backend that cannot provide it is not a drop-in (see [Known limitations](limitations.md)). The seam lives in `zuihitsu-core` so the wasm read replica can share it; only the durable file-backed backend, which needs the host filesystem, lives in the main crate.

### Materialised graph

SQLite, derived from the log. Tables project the log's structural state: `memories` (with the describer's per-memory staleness watermarks, `last_content_seq` and `last_described_seq`), `content_entries` (with the flattened occurrence columns, visibility, and the `superseded_by` supersession pointer), `entry_disputes` (the unresolved arbitration entries a read marks `disputed`), `tags` and `memory_tags`, `relations`, `links`, `memory_aliases` (former names, so a renamed memory is still found), `conversations`, `sessions`, `session_participants`, and `participant_identities` (the cross-platform identity map). A `meta` table holds the `graph_head` — the highest `seq` applied — which is what makes the graph self-describing about how far it has caught up. A single `memories_fts` FTS5 virtual table indexes name, description, and content for lexical search; it is populated with **public content only**, so a lexical hit needs no visibility filter and non-public content is reachable solely through its (visibility-gated) semantic vector. The graph DB can be deleted and rebuilt at any time without data loss; only its derivation logic is load-bearing, so its schema changes are drop-and-rebuild.

The catch-up machinery is uniform: `materialize_from` reads the log from `graph_head + 1` and applies each event, and the same code rebuilds a fresh graph, catches up a stale one, and heals a graph left half-applied by a crash. A content fingerprint over every projected table (in a canonical, layout-independent order, excluding the derived FTS index) lets a rebuilt-by-replay graph be checked byte-for-byte against its source, and lets a `VACUUM`ed snapshot be compared against the graph it was captured from.

Rebuilding from the log cures a corrupt or stale graph, but not a buggy materialiser handler. A wrong handler produces a clean, internally consistent graph that faithfully reflects a wrong interpretation of correct events, and rebuilding from `seq 0` reproduces the bug perfectly, because the log was never the problem — the code is. The consequence lands precisely on the visibility subsystem: a visibility-relevant materialiser bug is a silent leak that survives every rebuild. Replay is no defence here. The eval harness — the predicate and brief scenarios run against materialised state — is the backstop for materialiser logic bugs (see [Validation](observability-and-testing.md#validation-and-the-eval-harness)). This is part of why that backstop is load-bearing.

### Vector store

Separable, since the embedding model is a moving target: a standalone `sqlite-vec` database (a `vec0` virtual table with a cosine distance metric), or an external store. It sits behind a `VectorIndex` seam, so the in-memory backend serves tests and the sqlite-vec backend serves production.

*Embedding granularity:* both per content entry and per description are embedded. Entries are embedded so search retrieves at the granularity of what was actually said (the unit the predicate filters), and descriptions so thematic, summary-level recall works too; embedding is cheap enough that carrying both is worth the breadth. A vector's key encodes what it is — `entry:<entry-ulid>` for a content entry, `mem:<memory-ulid>` for a description — so a search hit maps back to the memory or entry it came from.

*Re-embed triggers* are correspondingly two: an entry vector is computed once on append and never again, because entries are immutable; a description vector is recomputed whenever the description regenerates. Steady-state embedding cost is therefore one vector per appended entry plus one per regen, not a whole-memory re-embed. Maintaining the index costs a model call, so it runs off the turn's hot path as a **reactive projection**: a background indexer consumes committed events (a `MemoryContentAppended` embeds that entry, a `MemoryDescriptionRegenerated` re-embeds the description, a `MemoryDeleted` drops the memory's description vector) and persists a `cursor` — the highest `seq` it has processed — in the index's own `meta` table, written *after* the batch's vectors are durable. So a restart resumes from the cursor rather than re-embedding the whole log, and a crash mid-batch re-processes that batch (an idempotent re-embed) rather than skipping it. A soft-deleted memory's entry vectors are left in place but become unreachable, since search resolves an entry hit through its now-absent memory.

Each vector carries the id of the model that produced it, stored at vector-creation (a `+model_id` partition column on the `vec0` table) rather than retrofitted, because retrofitting provenance onto already-written vectors is itself a full re-embed. In steady state every live vector shares one model, so a single stored `model_id` identifies the index's embedding space — what boot compares against the configured embedder to detect a swap — and it is what makes a mixed-embedding-space state detectable rather than silent.

Vectors do **not** carry visibility metadata; the index stores only the embedding, the key, and the `model_id`. Visibility is enforced at **search time**, not at the vector layer: a description hit surfaces its memory directly (a memory's description is synthesised from its public entries, so it needs no filter), while an entry hit is resolved to its content entry and passed through the same `visible(...)` predicate against the caller's present set before it can surface its memory (see [Visibility → Search](visibility.md#search-is-a-third-visibility-surface)). This is the counterpart to the public-only lexical index: the FTS index holds only public content and so needs no filter, whereas the semantic index is where non-public (attributed or private) content is reachable, gated at read time.

A full re-embed from the log — needed only on an embedding-model swap, which is itself a logged `EmbeddingModelChanged` migration (the model identity is environmental config, but changing it is a behaviourally-significant, recorded event) — is the single most expensive operation in the system, far costlier than rebuilding the graph. "Rebuildable" should not be read as "cheap": a full re-embed is a real operational event, not a casual one. It needs the crash discipline applied everywhere else, because a half-finished re-embed leaves two embedding spaces in one index and cosine across them is silently wrong (degraded rankings, not an error). On an embedding-model swap the boot path clears the index and re-embeds the whole log under the new model *before the server serves* — requests are refused until the rebuild completes, so search never runs over a mixed space, at the cost of unavailability during the rebuild.

### Snapshots

A snapshot is a checkpoint of the materialised graph, not of the log. The log is append-only and always retained in full, so there is nothing to snapshot there; what is expensive to rebuild is the derived graph. `VACUUM INTO 'snapshot-{seq}.sqlite'` produces an atomic graph file, its captured head zero-padded into the filename (so a lexical sort of the directory is a numeric sort by `seq`) and its `meta.graph_head` riding along in the copy, so the file is self-describing about the `seq` it holds.

Materialisation resumes from whichever derived state is furthest along: boot restores the latest snapshot **only when it leads the on-disk graph** (a fresh, deleted, or corrupt graph, whose head is below the newest checkpoint), copying it over the graph path so that `materialize_from` then replays only the tail — from the restored head to log-head — rather than the whole log. When the persisted graph already leads its checkpoints (the steady state), restore is a no-op and the graph catches up its own tail. This is the same tail-replay the boot and commit paths use everywhere (see [Commit and boot span two stores](#commit-and-boot-span-two-stores)). Branching an experiment is a file copy. Capturing a graph snapshot mid-commit is a storage-layer corruption hazard (see [Known limitations](limitations.md)), so the snapshot must be taken at a clean `seq` boundary.

Snapshots cover the graph; the vector index is a separate rebuildable projection whose re-embed is the far costlier operation treated above, so it is out of scope here. Snapshotting is **on by default** — the graph is always rebuildable, but a checkpoint turns a slow cold rebuild into a fast one, so the safe default keeps them; it is disabled with `[snapshots] enabled = false`. The clean-`seq`-boundary requirement is met by writing the snapshot under the graph lock a commit also takes, so a commit can neither be in flight nor interleave. The cadence is **activity-gated**: a background snapshotter checks every `check_interval_seconds` (default one hour) and snapshots only once at least `min_new_events` (default 20) have accrued since the last one, so idle periods never produce a snapshot; after each new one it prunes to the `keep` most recent (default 5), and never deletes the only checkpoint boot would restore from. An operator can also take one on demand — `zuihitsu snapshot` posts to the running server's `POST /control/snapshot`, which writes a checkpoint at the current head (or reports none when the graph is already snapshotted there) — e.g. before branching an experiment. The revert path discards any checkpoint captured past the revert point, so `restore_if_stale` cannot copy a future state back over the rebuilt graph.

### Schema evolution

The serialised payload carries no `version` field; a derived `payload.version()` accessor (`MergeProposed` is 3, `ConversationTurn` is 2, everything else 1) stamps a `version` column on the `events` table as recorded metadata. The materialiser dispatches on the payload enum variant, and a bumped version's field differences are absorbed inside that variant's arm by the `#[serde(default)]`/`#[serde(alias)]` evolution described above. Old events stay readable forever, and evolution is additive: a new capability adds a new event variant or a new field on an existing one, and old logs replay unchanged — no migrations.

### Soft delete

`MemoryDeleted` sets a `deleted` flag on the projection; contents are preserved. The flag filters the memory from agent-facing reads, search, briefs, and `same_as` traversal, and hides links touching it from agent traversal. The memory and its links remain in the log and the materialised tables for replay, audit, and `BeforeAfter` anchor resolution (which reads contents directly, bypassing the filter). Deletion is soft and auditable; the data is never destroyed.

### Commit and boot span two stores

The event log and the materialised graph are separate databases, so a block's commit is not one atomic write. Define it precisely: commit appends the block's buffered events to the log, then applies them to the graph, under the block's held locks. The log append is the durable commit point; the graph apply is replayable derived work. That framing makes the two-store problem tractable: rather than making both stores atomic, one is authoritative and the other is reconstructable from it. Two consequences fall out:

- *In-block reads are an overlay, not a plain graph query.* Within a block, `dave:append("X")` then `dave:get()` must see "X" before commit, but the buffered event is not in the graph (or the log) yet. So an in-block read queries the materialised graph and overlays the block's pending buffered effects, applying supersession and the same `visible(...)` predicate as any other read — visibility holds inside a block too. The overlay is real, fiddly code, not a free property of buffering.
- *Boot reconciles graph-head to log-head.* If the process dies in the commit window — events appended to the log, not yet applied to the graph — those committed events are not reflected in the graph. This self-heals only if boot re-materialises the tail before serving, rather than trusting the graph as-is: it restores the latest snapshot when that snapshot leads the on-disk graph, then replays from the resulting `graph_head + 1` to log-head. It is the same machinery as recovering a stale or corrupt graph: the graph is always derived, so catching it up is just replay of the tail. Commit itself is `append` to the log, then `materialize_from` to bring the graph up, both under the block's held graph lock — the log append is the durable commit point, the graph apply the replayable derived work.

### Faithful replay

The system replays **faithfully**: it reconstructs exactly what happened, materialising from events using the stored outputs of past inference and execution. Descriptions, arbitrations, and `LuaExecuted` results are already in the log as result events, so neither the model nor the Luau VM is re-invoked. The behaviour is a pure function of the event log modulo declared nondeterminism (ULID minting, wall-clock stamps), so replay through the in-memory backends exercises exactly that function. This is what normal boot and time-travel use, and it is what the whole "rebuild from the log" story rests on. Every inference-derived event also records a `produced_by` provenance stamp (the model and template version behind it), so an older event still names the body it was generated under even after templates move; this metadata is recorded for provenance and available to a regenerative replay.

Some events are **recorded observations** rather than materialised state: timestamps (`recorded_at`) and the model-interaction record (see [Observability](observability-and-testing.md#observability)). The materialiser ignores them, so the rebuilt graph is byte-identical with or without them, and faithful replay reproduces their non-deterministic content (a model's reasoning, a call's latency) verbatim because it reads them rather than recomputing. They are part of *what happened*; they are inert for *what the state is*. A belief arbitration is a deliberate in-between: its reconciling statement is a recorded observation, but its *unresolved* competing entries now project into the graph (the `entry_disputes` table) so a read can mark a contested fact `disputed` (see [Write path → arbitration](write-path.md#coalesce-then-regenerate-once)), so the event is materialised like a description rather than ignored like the model-interaction record.
