# Conversations and briefs

## Conversations and contexts

A *conversation* is a durable, addressable room: the agent meets the same room again and again and remembers it. The system distinguishes two levels — the durable conversation and the session within it — and ties both to memory.

### The locator keys the room

A platform client supplies a `ConversationLocator` — `(platform, scope_path)`, both `SmolStr`s the client chooses (`ConversationLocator` in `crates/core/src/ids.rs`). `platform` is a short config key naming the integration; `scope_path` is that platform's hierarchical address of the subcontext, opaque to the server. Two locators name the same room exactly when both fields match. A connector for a chat platform maps its room hierarchy into `scope_path` by its own convention — a Discord guild channel might become `(discord, guild/42/chan/leads)`, a thread appending its id, a DM keying off the channel — so a DM, channel #a on server X, channel #b on server X, and channel #a on server Y resolve to four distinct conversations even though they share one integration and may share participants.

The only clients are the operator's own console and the generic `POST /platform/message` HTTP endpoint that any external connector delivers through. The console drives the control paths under the locators `(operator, imprint)` for the imprint interview and `(console, lua)` for a console Lua run (`Control::imprint` and the console-run action in `src/instance/control/actions.rs`); a reserved `direct` platform key (`DIRECT_PLATFORM` in `crates/core/src/ids.rs`) marks a participant identity as arriving through that console, where the sender is operator-chosen and so a handle match carries operator authority (see [Cross-platform identity](data-model.md#cross-platform-identity-operator-asserted-or-adjudicated)). An external connector — Discord, Slack, or any other — posts to `/platform/message` with a locator of its own devising.

The server maps a locator to a stable internal `ConversationId` (a ULID) the first time it is seen and records the pairing on a `ConversationStarted` event, so the same locator next week resolves to the same conversation, giving continuous memory of that room (`resolve_or_mint_conversation` in `src/memory/identity.rs`). Resolution reads the graph's locator index; a `conversation_for_locator` miss mints a fresh id.

### Durable conversation vs. session

A chat room is persistent and spans idle gaps, so two units diverge. The conversation (locator-keyed) is the unit of memory continuity. A session is a bounded window of activity within it — opened on first activity, when activity resumes after a quiet period (past `idle_gap_seconds`, thirty minutes by default), or when the live buffer crosses a token budget (see [Compaction](#compaction-token-triggered-re-segmentation) below) — and is the unit that freezes a brief and anchors the prefix cache. Session boundaries are recorded (`SessionStarted` / `SessionEnded`), never recomputed at replay, so a resumed session reads back identically.

On a cold start (a restart, or a passive graceful exit that left a session open), the next message resolves the dangling session lazily: within the idle gap it resumes it untouched — the prompt prefix is byte-identical, so the serving cache survives the restart — and past the gap it closes it with a flush before opening a fresh one (`ensure_session` in `src/instance/session/lifecycle.rs`).

Turns belong to sessions, and a conversation accumulates many sessions over its life. `ConversationStarted` fires once when a room is first seen; `SessionStarted` fires per activity window and is what the brief-freeze machinery keys off.

### Present set is per-session, supplied by the client

The server does not track a platform's full channel roster; it knows who is participating in the current session, which the platform client reports. Membership drift across a conversation's life is just different present sets across its sessions, and mid-session arrivals use `ParticipantJoined`. The agent reasons about who is present, not who is a member.

### Transcript references

A participant often wants to point back at a specific earlier moment — "remind me what we decided here" — rather than re-describe it. A **turn reference** names one `ConversationTurn` by its ULID, carried either as the canonical `[turn:<ulid>]` token or as the `?turn=<ulid>` query parameter of a console deep-link. Both forms parse to the same turn through one module (`turn_ref`), which is the whole definition of what counts as a reference: the agent's resolver, the console's renderer, and the console composer's send-time normaliser all read the same parser, so the syntax cannot drift between them. The module is dependency-light (ULID parsing, no URL crate) so it compiles to wasm and drives the console through the same code the agent runs. `convo.turn(id)` resolves a reference to that moment plus a small window of the turns around it in its session, each turn carrying a ready-made `ref` so the agent cites a moment by copying the canonical token rather than assembling syntax by hand.

Resolution obeys one **audience rule** for every conversation: a turn resolves iff every member of the current present set was in that turn's audience (`resolve_turn` in `src/agent/turn/resolve.rs`). A turn's audience is derived from the log by one forward pass — its session's `SessionStarted.participants` plus every `ParticipantJoined` for that session up to the turn's seq — so the check is "was everyone here also there then." Both sides of the comparison are first mapped through their `same_as` class, so an operator-merged cross-platform identity (`person/maya@direct` merged with `person/maya@discord`) reads as one person: a present member is admitted when any of its class siblings was in the audience, matching how class-wide reads already treat a merge (see [Visibility](visibility.md)). A member whose class cannot be read falls back to its raw id — the strict pre-merge behaviour, never a looser one.

The window is clamped to the focal turn's session and each neighbour is re-checked against the same rule, so a mid-session join never widens the window past what the present set shared. The audience rule is not same-room scoped: a solo DM resolves any turn the requester attended, in any room, and a two-person DM resolves turns whose audience included both — the reference travels with the people who were there, not with the room it was said in.

The refusals are two, deliberately distinct. A reference to a real turn whose audience the present set was not all party to fails with a teachable warning that names the audience problem and points at memory — the visibility-filtered channel — rather than the transcript: *recall it through memory instead of paraphrasing what was said.* A reference that names no turn at all is the ordinary not-found. The distinction confirms, for a cross-room id, that the id maps to *something* — a small, deliberate loosening justified by ULIDs being unguessable: holding one means you were there, or someone who was there gave you the link. Within a resolved window no content visibility filtering is applied, because the audience rule already guarantees the whole present set was party to every turn shown — resolving it opens no visibility surface that presence did not already open.

### Compaction (token-triggered re-segmentation)

The live buffer — `ConversationTurn`s appended as a suffix to the frozen prefix — is the one accumulating surface with no inherent bound, and a busy room that never goes idle would otherwise walk straight into the context limit. The bound is native to the session model. When a turn's peak prompt crosses a soft token budget, the session ends *now*, and the next message re-segments: it opens a fresh session, re-freezing a brief against the current present set, which also folds in joins and leaves at the boundary and re-grounds visibility, consistent with the join semantics (`route_message` and `end_session_for_compaction` in `src/instance/platform.rs`). The cut is deliberately lazy — the ending is recorded at once so the flush runs on the hot path, but the re-open waits for the next message rather than eagerly rebuilding a brief no one is waiting on. The trigger reads the turn's real `prompt_tokens` when the backend reports it, and falls back to a coarse `chars / 4` estimate over the buffer plus the inbound message otherwise (the in-memory and no-OpenAI builds); the estimate omits the system prefix, so it under-counts, and only the real usage is authoritative. 

An operator can also force a seam through the exact same path (`Platform::force_compaction`), stating "cut here" directly rather than sizing a budget so the organic trigger happens to fire. 

A deferred turn — one where the model proved unreachable — skips the compaction check entirely: the flush could not run anyway, and the buffer gained no agent turn. And if the model goes down between the reply and the compaction flush, the reply is still delivered and the session is left open for the next message's recovery rather than being turned into an error.

The budget is not the hard context limit — it is a fraction of the model's stated context window, `COMPACTION_BUDGET_FRACTION` (0.8) of it, leaving a fifth of the window as headroom for the system prefix, the reply, and the next session's carryover seed. The window itself is operator-stated config (the inference API does not report it), so the budget is derived from it once at genesis (`compaction_budget_for` in `crates/core/src/settings.rs`, applied in `src/agent/genesis/mod.rs`) and baked into settings; an instance created without a configured context length keeps the default `token_budget` of 24,000, and an explicit settings override always wins. Because a session is a view over the log and not a store, nothing durable is at stake; the design effort is in carrying continuity across the cut without putting a model-authored artefact into the buffer:

- **Pre-compaction flush (budget-gated).** Before the cut, if the ending session was substantive enough to have accumulated working state — at least `flush_min_turns` (four by default) turns in the bounded buffer — the agent gets one turn whose explicit job is to write to memory anything worth keeping that it hasn't already, with the same visibility discipline as an ordinary turn. A low-activity session — say one that crossed the budget via a single large paste — falls below the threshold and is skipped, so the hot-path model call is paid only when there is state worth saving. The flush summarises into memory, where durable things belong, rather than into the buffer, so no recap lives only in context. It is an ordinary turn (`ConversationTurn` + `LuaExecuted`), fully logged and replay-trivial. It runs **before** `SessionEnded`, so a flush that fails (the model went down) leaves the session standing for the next message's recovery rather than dropping its state; the same `flush_and_end` path also backs the idle-sweep and cold-start-recovery closes, not only the budget compaction. The cost, named: it runs the model on the hot path at maximum context, the worst moment for latency, and the budget gate is what keeps that cost from being paid when there's nothing to flush.
- **Checkpoint flush (mid-session, watermark-scoped).** The same flush turn, decoupled from session end: without it, a parallel conversation learns nothing of this one until it goes idle or compacts, since working state reaches memory only at the seam. A background sweep runs the flush mid-session and leaves the session open — no `SessionEnded`, no carryover, no brief rebuild; the flush turn simply rides the live buffer (its terminal is silent, so participants see nothing), and its seq becomes the session's **flush watermark**, derived from the log (the last turn whose `produced_by` names the Flush template, else the session's start) rather than held as mutable state. The flush prompt covers only the buffer past the watermark, so repeat checkpoints never re-flush the same turns. Three gates keep the model call honest (`checkpoint_delta` in `src/instance/drivers.rs`, tunable through `CheckpointSettings`): *substance* — the delta past the watermark carries at least `min_delta_chars` of participant- and agent-turn text (system turns like join briefs and drained wake-ups do not count toward it); *cooldown* — at least `cooldown_seconds` have passed since the watermark turn was recorded, or since the session opened if it has never flushed, so checkpoints never thrash; and *audience* — some other conversation has a live session whose last activity is at or after the watermark's anchor, since with a single live conversation the only reader of the tail is the conversation itself, which already has it in the buffer. The whole sweeper can be turned off with `checkpoint.enabled`.

  A cheap gate pre-check runs without the lifecycle lock, so the lock is taken only for a real candidate and the gates are re-validated under it (a message since the candidate list was captured may have closed or already flushed the session). Each flush runs under the conversation's lifecycle lock, so a message arriving mid-flush waits on its own conversation's delta-sized flush, never on another's.
- **Raw transcript carryover (character-budget).** The new session is seeded with the tail of the old buffer, filled backward from the cut up to `carryover_char_budget` characters (4,000 by default) — as many recent turns as fit, adapting to message size rather than a fixed count. The newest turn is always carried even if it alone exceeds the budget, so the immediate thread never drops entirely. The tail is staged from the ending session's *own* turns (those at or after its `SessionStarted`), not the whole read buffer, so the carryover point advances into the current session with each seam rather than sticking at the original one. Deterministic, verbatim, no model. On the next open the carried tail is bounded to the same char budget on read, so it cannot grow across seams.
- **Working-set carryover (deterministic, platform-derived).** The new brief is composed normally, then augmented with a working set assembled deterministically, with no relevance judgement at rebuild time. (1) Touch-derived: every memory ID the ending session read or wrote, taken from the per-block `touched` sets on its `LuaExecuted` events (the lock set the concurrency layer already computes). Reads emit no other event, so the touched set — not a scan of `script` or `result` — is the source, and the read half is the more valuable half, since the agent looked something up because it was relevant. The flush's own reads and writes land in the touched set too, so anything it deliberately rescued survives. (2) Recency-derived: the normal brief's `recent_facts`, already free. Both are re-filtered through `visible(...)` against the new present set. There is no third, agent-managed flag source: session-lifetime flags on the semantic graph are fragile, and the touched set already captures what matters. The agent controls what survives during the session — by touching memories and writing in the flush — not by a magic call at the seam, which keeps the rebuild deterministic.
- **Regen-to-completion before the new brief.** The flush- and touch-derived memories are exactly the ones the new session re-surfaces, and the flush just wrote to some of them, whose descriptions now lag because regeneration runs as a background catch-up, not inline in the flush. The post-compaction brief would otherwise read off stale prose for precisely the memories flagged most important. The fix is the guard already built for participant joins (see [Write path → Freshness before a brief](write-path.md#freshness-before-a-brief)): force the description catch-up to completion before composing the new session's brief, narrowed to exactly the brief's own read set — the present set, the room's context memory, the carried working set, and the agent's `self` — so a session open pays only for the descriptions its brief reads, not a whole concurrent backlog another turn left stale.

The honest seam: the *ambient transcript* — everything said but never recorded, referenced, or flushed, and older than the carryover budget — is lost from context. It remains in the log forever, just not in front of the agent. This is the right loss, since un-acted-on chatter is what you want to shed, and the flush is the deliberate chance to rescue anything that matters.

The genuine residual, even with the flush, is in-flight reasoning: synthesis the agent was mid-way through that never became a memory or a turn. A hard cut loses it, and the flush helps only insofar as the agent can dump working state to memory in that turn. This is named in [Known limitations](limitations.md).

### Contexts are first-class memories

Each durable conversation has a corresponding `context/*` memory, minted eagerly on first activity (like a person stub) alongside the `ConversationStarted` event that pairs it to the conversation (`resolve_or_mint_conversation` in `src/memory/identity.rs`). It is minted under a provisional locator-derived name — `context/<platform>:<scope_path>`, e.g. `context/discord:guild/42/chan/leads` — which the agent or the operator renames to a friendly handle (`context/acme-leads`) later; the platform supplies no display info at mint. The locator resolves to it, so the agent can look it up and reason about it: what this room is, who tends to be here, and whether things said here are said in confidence.

A room's confidentiality is carried by a `#confidential` tag on the context memory. The tag is the load-bearing signal because tags are memory-level and present-set-independent: they are visible regardless of who is in the room. A plain content entry would be the wrong home, because a non-person memory has no subject, and although such entries default `Public` (see [Visibility → Defaults](visibility.md#defaults-at-write-time)), the always-visible guarantee belongs on the tag, not on a fact that could be marked private and then vanish when its teller is absent.

The tag is set by the agent from conversational cues ("keep this in here," a private DM being implicitly confidential) or by the operator through the console, and supporting facts may accompany it. Removal is asymmetric: only the operator may remove the tag, because removal retroactively re-renders every aside recorded under it (see [Trust model](trust-and-authority.md#trust-model)). A context memory is itself a non-person memory and follows that visibility profile: [teller-gated](visibility.md#what-privatetoteller-actually-promises) entries, no [subject-guard](visibility.md#the-read-time-predicate).

### `told_in` provenance

Every `MemoryContentAppended` stamps the `ConversationRef` it was told in — the conversation, and the turn when there is one (`MemoryBlock::new` in `src/memory/memory_block/mod.rs`). The turn is absent (`None`) on the operator-console and genesis paths, which write outside any participant turn; a `LinkCreated` carries the same `told_in` provenance, mirroring content entries. This is provenance, not a gate: it is deliberately not part of the visibility predicate. The `#confidential` check reads from the *resolved current room* context, not from a stored entry's `told_in`: the block resolves whether the room it is writing in is `#confidential` at open time (`confidential_context` in `MemoryBlock::new`). Gating on `told_in` instead would fold the originating context into the `visible(...)` predicate, reopening the audience-gating we closed.

What it buys is judgement with memory. The agent can resolve an entry's `told_in` to its `context/*` memory and learn the room was confidential, and it knows the confidentiality of the room it is currently in (the current context memory is in the brief — see [Contextual briefs](#contextual-briefs)). So an aside told in the private team channel can be treated as confidential when the agent later finds itself in a different room, and new asides in a room known to be confidential are marked private by judgement. `told_in` is recorded as provenance; the `visible(...)` predicate does not read it.

## System prompt

The frozen system prompt (frozen per session — see [Conversations and contexts](#conversations-and-contexts)) is assembled by `system_prompt::assemble` into a fixed sequence of sections, each drawn from a source with its own provenance, and a builder needs the whole manifest in one place rather than inferring it from scattered references. The sections, in emission order, are: the **scaffold**, the agent's **identity**, the **API description**, the **runtime vocabulary**, the **contextual brief**, and the **declared time**. Each is recorded as a labelled span over the assembled text (`PromptSectionKind`), so a consumer — the context debugger in particular — can slice the prompt back into its contributions; the spans tile the whole text with no gaps. The scaffold and the time are always present; identity, the API description, the vocabulary, and the brief each drop out cleanly when empty.

**The scaffold** — a versioned `PromptTemplateRegistered` template (see [Initialisation](lifecycle.md)), emitted first with no header of its own. It carries the durable, agent-owned framing: how the agent operates, not who it is. The persona is layered in separately as the **identity** section (headed `# Who you are`), drawn verbatim from `self`'s content *entries* (the immutable, append-only charter), never from its regenerable description, so the authored voice cannot drift even as the self evolves through appended self-observations. The description stays a lossy summary, used for search and compaction, and is deliberately not the source of the voice. The scaffold's body is a preamble followed by a sequence of one-concern dotpoints, and it covers:

- *How it operates:* that it acts by emitting Luau through the `run_lua` tool, that a turn is a loop of steps, that memory persists across sessions while the in-block scratchpad does not, and that it talks with several participants who do not all see the same things.
- *The namespace ontology and how to query it:* what each namespace holds (the prefixes are sourced from `Namespace` so the scaffold and the handle-minting code cannot drift), and that merged identities are read through the canonical handle (`person/marcus`, not a `@platform` stub) so the agent doesn't look in the wrong drawer and miss facts.
- *Informed creation — recognise before you create:* that a fact's home is found by searching for the existing memory before minting a new handle, so a guessed name doesn't split one topic or event across two memories, and that a search hit's salient relations are there to confirm the match.
- *The `agent`-teller convention* for recording its own observations and inferences.
- *The compartmentalisation principle and the teller-private marker semantics* (see [Visibility](visibility.md)).

The **declared current time** rides in its own trailing section (headed `# Current time`), reading "The session begins on \<datetime\>." — frozen to the session's start, not the live clock, so every turn in the session sends an identical system prefix and the serving layer can reuse its prefix cache. Each subsequent turn carries its own recorded time in its per-message stamp, so "now" stays current without rewriting the prompt (see [Time](time.md)).

**The API description** (headed `# What you can do`) — build-derived, not versioned (see [Lua API](agent-loop.md#lua-api)): the catalogue of callable functions, including the connected MCP servers' projected tools, rendered from the running binary so the agent always knows what it can call. It is prefixed with a short notation legend explaining the `<memory>:method(...)` convention, so the model calls a handle method as `handle:method(...)` rather than pasting the placeholder literally.

**The runtime vocabulary** — a distinct section following the API description, composed by the caller from graph reads rather than from the binary, because tags and relations are runtime data the agent coins as it goes. It renders the current tag vocabulary (`# Tags`, each line `name — purpose (N uses)`) and the registered link relations (`# Relations`, each line `name / inverse — from-to[, symmetric][, reflexive]: description`), so the agent knows which labels `mem:tag` and `links.create` accept and which are already in use. Either subsection is omitted when empty.

**The contextual brief** (headed `# What you know right now`) — composed fresh per session (see [Contextual briefs](#contextual-briefs) for its structure and budget). The system prompt section deliberately does not restate the brief's internals; it only records that the brief is the section that varies per session and present set.

The scaffold's dotpoints and the API description are both feature-gated by the instance's `InstanceFeatures`, in lockstep with the Lua registration: a disabled feature drops the dotpoint that teaches its practice and the reference entries that describe its functions, so the prompt never teaches or advertises something the runtime rejects (see [Instance features](../CONTRIBUTING.md#instance-features) in `CONTRIBUTING.md`). The feature-gated dotpoints are baked into the scaffold at genesis and read back verbatim at turn time; the API description is re-rendered from the running binary's feature set each turn.

Stated as an acceptance check: a correctly assembled system prompt orients the agent to its identity and persona, how it operates, the namespace ontology and canonical-handle querying of merged identities, the `agent`-teller convention, the injected API description, the runtime tag-and-relation vocabulary, the compartmentalisation principle and marker semantics, the brief block, and the declared current time.

### On replay: the prompt is faithfully replayable

The frozen prompt is an input the agent saw, so it is captured — and captured two ways. The whole assembled `system` string, together with its section spans and a SHA-256 digest of the request, is recorded verbatim on the first `ModelCalled` of each phase when observability capture runs at `Full` (see [Observability](observability-and-testing.md#observability)); the digest lets a reconstruction be checked against the call actually sent. Separately, the composed brief is recorded verbatim on `SessionStarted`, independent of capture level. Faithful replay reads the recorded `system` string byte-for-byte, with no reconstruction at all.

A reconstruction from the log alone — the regenerative path — reassembles the prompt from its sources: the scaffold from its versioned template, the identity from `self`'s entries, the vocabulary from the materialised graph, the brief from `SessionStarted`, and the frozen session-start time. All of these are in the log and reproduce exactly. Only the **API description** is not stored: it is build-derived, re-rendered from the running binary rather than snapshotted, so a regenerative reconstruction reflects the current binary there — byte-identical to the original only when the API has not changed since. The brief specifically must be captured rather than recomputed, because recomputing it would run deterministic composition over *current* state, not the state at the time; capturing it on `SessionStarted` is what closes that gap for the regenerative path. Faithful replay, reading the captured `system` verbatim, has no such gap.

## Contextual briefs

At session start, `brief::compose` deterministically assembles the agent's hot context into a block that becomes the `# What you know right now` section of the system prompt and is frozen for the session's duration (see [Conversations and contexts](#conversations-and-contexts)). Composition reads the graph but runs no model and makes no relevance judgement beyond a deterministic recency ranking, so a leak into a brief is a mechanism bug catchable without inference. This replaces any explicit pinning.

Before the brief composes, a session open forces the description catch-up over the brief's own read set — the present set, the room's context memory, the carried working set, and `self` — so the brief never reads stale prose for a memory a prior turn just wrote (see [Write path → Freshness before a brief](write-path.md#freshness-before-a-brief)). The catch-up is narrowed to that read set rather than the whole backlog, so a session open pays only for the descriptions its brief reads.

### Composition

The composer emits these sections in order, each dropping out when it has nothing to show:

1. **Self brief** (`# You`). The `self` memory rendered in the per-participant body shape below. `self` has no participant-subject, so its `PrivateToTeller` entries are teller-gated only: a private aside told about the agent surfaces whenever its teller is present, governed by the marker among co-present third parties, and suppressed from any room not containing the teller.
2. **Current room** (`# Current room: <room>`, suffixed `(confidential)` when the context memory carries the `#confidential` tag). The `context/*` memory for this conversation, with its confidential flag riding along at the memory level (not teller-gated), so the agent walks in calibrated to where it is and treats new asides in a confidential room as private by judgement. Its description follows on the next line when it has one.
3. **Present participants** (`# Present`), one full block per participant within the present-set cap, ranked by recency; the tail collapses to a name-only line (see [Present-set cap](#present-set-cap)).
4. **Active threads** (`# Active threads`). The working set carried across a compaction seam — the memories the ending session touched — re-surfaced so the new session does not lose the thread. Each is rendered in the same body shape, so its facts are re-filtered through `visible(...)` against the *new* present set (an aside about a now-present subject is suppressed). Self, the current room, and present participants are skipped here to avoid duplication, and a thread whose body filters to nothing is dropped. Empty when the session opened without carryover (see [Compaction](#compaction-token-triggered-re-segmentation) → working-set carryover).
5. **Upcoming** (`# Upcoming`). Near-future calendared occurrences within `upcoming_window_days` of the session start, soonest first, capped at `max_upcoming_items`, so the agent organically raises them (see [Calendar](time.md#calendar-as-a-view-over-memory)). Each line is `- <day>: <name> — <text>` and passes the same `visible(...)` predicate as any entry — a private aside about an absent person carries its marker, one about a now-present subject is suppressed.

### Per-participant body shape

Each memory's body (self, a present participant, an active thread) renders as a `## <name>` header followed by:

```
## marcus
<summary>{description}</summary>
<recent_facts>
- {visible fact} {markers}
</recent_facts>
<relationships>
- {relation}: {subject} {marker} — "{neighbour's latest visible fact}" {markers}
</relationships>
```

The header is the memory's canonical name; there is no `id` attribute, and the block is line-oriented markup, not a self-closing XML element. `<summary>` is emitted only when the memory has a non-empty description, `<recent_facts>` only when at least one visible entry survives, and `<relationships>` only when at least one visible edge survives.

The audience-specific precision lives in `recent_facts`: the memory's last `recent_facts` (default 8) content entries that pass the `visible(...)` predicate against the present set, in commit order, oldest first — deterministic, with no model judgement in the filtering. Each surviving fact carries its markers beside the text (rather than baked into one string, so a structured consumer can style them quietly): the teller-private `[via …]` / `[private · …]` provenance marker when the entry is non-public, then a staleness marker when the entry has decayed on a high-volatility memory (see [Time → Recency and volatility](time.md#recency-and-volatility)).

`relationships` lists every *live, visible* edge touching the memory — a soft-deleted neighbour is skipped, and each edge passes `link_visible(...)` against the present set. An `Attributed` or teller-private link carries its provenance marker (`[via Erin]`) appended after the line, since a link has no text body of its own. Each edge also carries its neighbour's most recent entry visible to this audience, clipped to 200 characters, as an inline spoke fact — so the relationship line is not spoke-blind: a fact filed on a linked hub (`Priya hosts book club at her place`) reaches the person's brief through her `participates_in` edge rather than being invisible for living on the neighbour. The spoke fact runs the *exact* same visibility predicate as a recent fact, so a confided aside on the neighbour never rides into a brief for an audience that may not see it.

"Who created you?" is answerable from the system prompt directly: `created_by` is a structural, public link and appears in `self`'s relationships regardless of the description.

### Mid-conversation joins

On a join: emit `ParticipantJoined`, build only *that* participant's brief (filtered against the now-present set, which includes the joiner — so the subject-guard suppresses asides about them, since a subject joining closes the dangerous direction), and record it as a `System` turn at the join point rather than rebuilding the frozen prompt, which preserves the prompt cache up to that point. The `System` turn carries the rendered markup as its `text` (the prompt build reads turn text verbatim, so the agent path is unchanged) and the structured `Brief` alongside it, so a structured consumer (the console) can render a proper entrance rather than parsing the markup back out. When a model is available the joiner's description is caught up first; the modelless `/platform/join` path composes off the current prose — a slightly stale join-brief beats refusing the join.

### Size budget

Each memory body is bounded by the `recent_facts` count (default 8); there is no per-block token truncation (`brief.token_budget`, default 2000, is advisory and unenforced), and the composer lists every live, visible edge without ranking or a cap. The expensive synthesis happens at description-regeneration time, not here.

### Present-set cap

The per-block `recent_facts` bound governs the depth of each participant's block, but the number of participants is itself unbounded — a 30-person channel would produce 30 full blocks. So the present set is ranked and capped by `present_set_cap` (default 10): participants are ordered by the recency of their latest asserted entry across the merged identity (with the memory id as a deterministic tie-break), the first `present_set_cap` get full blocks, and the tail collapses to a bare `- <name> (present)` line. This is the participant-axis analogue of bounding facts within a participant. It is a distinct bound from compaction, which bounds the conversation buffer, not the brief: re-freezing a brief against a large room reproduces the same block count, so the buffer trigger does not address this vector. A token-triggered re-segment opens a new session, which recomposes the brief and re-applies the cap against the new present set.

### Invariant: the cap never narrows the present set for visibility

The cap governs who gets a full brief block, nothing else. `presence(...)`, and therefore the entire `visible(...)` predicate, always resolves against the full, uncapped present set — `compose` passes the whole `present_set` to every `visible(...)` and `link_visible(...)` call, above and below the cap. A tail participant collapsed to name-only is still present: the subject-guard must fire against them, an excludee among them must suppress, and so on. Reading "cap the present set" as "the predicate evaluates against the capped set" would reintroduce a leak in exactly the high-population rooms the cap exists to serve, so the two sets are kept distinct by construction. The cap is a brief-allocation device applied after the predicate has already been evaluated against everyone present.
